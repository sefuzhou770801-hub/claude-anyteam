## §1 Knowledge articulation

- From training + current `openai/codex` docs, App Server is a JSON-RPC interface used by rich clients; the core persisted objects are **thread**, **turn**, and **item**. Verified in `codex-rs/app-server/README.md`: https://github.com/openai/codex/blob/main/codex-rs/app-server/README.md
- The handshake is `initialize` once per connection, then `initialized`; requests before that fail with `"Not initialized"`. I reproduced that locally on `codex-cli 0.122.0`.
- Session continuation is a **first-class primitive**, but only via explicit thread lifecycle methods:
  - `thread/start` = new conversation
  - `thread/resume` = reopen an existing stored conversation by `threadId`
  - `thread/fork` = branch from an existing stored conversation into a new `threadId`
- README language is explicit that “if you’re continuing an existing conversation, call `thread/resume` with its ID instead,” and “if you want to branch from an existing conversation, call `thread/fork`.” I do **not** recall any first-class `thread/start(resumeFrom=...)` style shortcut, and local probing below supports that.
- Persistence semantics are file-backed for non-ephemeral threads: responses include a `thread.path` under `~/.codex/sessions/YYYY/MM/DD/rollout-...jsonl`. However, that file is not guaranteed to exist immediately after `thread/start`; materialization happens once rollout-backed history exists.

## §2 Empirical probe results

- Environment: local `codex-cli 0.122.0`, `codex app-server`, JSON-RPC over websocket and stdio.
- `thread/start` before `initialize` → `-32600 Not initialized`. Interpretation: normal handshake enforcement.
- `thread/clone`, `thread/continue`, `session/load`, `conversation/restore` → `unknown variant ... expected one of ...`. Interpretation: no hidden continuation verb under those names.
- `thread/resume {}` and `thread/fork {}` → `missing field threadId`. Interpretation: both are real methods and continuation/forking are explicit-ID operations.
- `thread/start` with speculative params:
  - `resumeFrom`
  - `parentThread`
  - `continueFrom`
  - `priorThreadId`
  - `seedFromThread`
  - `initialContext`
  all returned **success**, but each created a brand-new thread with `forkedFromId: null`.
- Interpretation of the above: these params are **ignored**, not honored. They are not validated as unknown-field errors, so a naive client could falsely believe inheritance occurred.
- `thread/start` with `sandbox:"workspaceWrite"` failed (`unknown variant 'workspaceWrite', expected 'read-only' | 'workspace-write' | 'danger-full-access'`). Interpretation: transport enum names are kebab-case even when responses serialize `workspaceWrite`.
- `thread/start` with correct params and `persistExtendedHistory:true` succeeded, returning a new `thread.id` and `thread.path`.
- Fresh thread immediately after `thread/start`:
  - `os.path.exists(thread.path)` → `False`
  - `thread/read(includeTurns:false)` → success
  - `thread/read(includeTurns:true)` → `thread ... is not materialized yet`
  - `thread/resume(threadId)` → `no rollout found for thread id ...`
  - `thread/fork(threadId)` → same `no rollout found ...`
- Existing materialized thread from `thread/list`:
  - underlying rollout file existed on disk
  - `thread/resume(threadId)` succeeded and returned prior turns/items
  - `thread/fork(threadId, ephemeral:true)` succeeded and copied prior turns into a new thread
- Two websocket clients resuming the **same existing thread id** against one shared app-server both succeeded. Interpretation: multi-client access to one stored thread works; this is not the same as cross-thread memory inheritance.

## §3 Workaround hypotheses tested

- Hypothesis: “Maybe a new thread can inherit another thread by passing hidden `thread/start` params.”
  - Test: sent six speculative inheritance params.
  - Result: all ignored; new blank threads were created. No continuation primitive found here.
- Hypothesis: “Maybe just knowing `thread.path` under `~/.codex/sessions/` is enough to resume.”
  - Test: fresh thread returned a path, but the file did not exist yet; `thread/resume` and `thread/fork` both failed until a real rollout-backed session existed.
  - Result: path alone is insufficient; continuation depends on a materialized rollout file / stored thread state.
- Hypothesis: “Maybe config has `initial_thread_id` or similar hydration knob.”
  - Test: `config/read` plus grep of generated schema and local `~/.codex/config.toml`.
  - Result: no `initial_thread_id`, `initialThreadId`, or equivalent continuation-at-start setting found.
- Hypothesis: “Maybe multiple clients can share backend session state if they multiplex one app-server.”
  - Test: two websocket clients both called `thread/resume` on the same existing thread.
  - Result: yes for the **same thread id**; no evidence that distinct thread ids share context implicitly. The closest supported pattern is “resume same thread” or “fork existing thread.”

## §5 Summary

- **Does a continuation primitive exist?** Yes, but not in the cross-thread-autoinherit sense. The supported primitives are `thread/resume(threadId)` for continuing the **same** thread and `thread/fork(threadId)` for creating a **new** thread with copied history.
- **Can one new thread inherit another thread’s session state via `thread/start` params?** I found no supported primitive. Speculative params are silently ignored and produce fresh threads.
- **Closest thing to v7.2 cross-task memory on one adapter:** store the source `threadId`, then explicitly call `thread/resume` (same thread) or `thread/fork` (branched thread). Without that explicit ID handoff, there is no first-class session-continuation-at-start feature.
- **Important caveat:** continuation only works once the source thread is actually materialized on disk / persisted; a just-started thread with no rollout file cannot yet be resumed or forked.
