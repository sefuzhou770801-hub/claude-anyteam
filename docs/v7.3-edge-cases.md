## 1) `ephemeral=False` disk footprint

`app_server_invoke` already sets non-ephemeral threads (`src/codex_teammate/codex.py:702-718`) so each task leaves a rollout file under `~/.codex/sessions/YYYY/MM/DD/`.

I measured a 5-task chain (1 `thread/start` + 4 immediate `thread/fork`s) with one turn per task via `/tmp/v73_footprint_probe.py`. I used the five known rollout paths directly rather than a naive before/after day-dir delta, because a concurrent acceptance run created an unrelated extra file during the probe window.

| Task | Rollout size |
|---|---:|
| 1 | 12,972 B |
| 2 | 17,655 B |
| 3 | 22,491 B |
| 4 | 27,158 B |
| 5 | 30,728 B |

Totals for the 5 chained tasks:

- **111,004 bytes total**
- **22,200.8 bytes/task average**
- **~21.7 KiB/task**

So the footprint is comfortably negligible relative to the “<1 MB/task” bar.

Observations:

- Each task produced one rollout JSONL file; I saw no extra sidecar file per thread.
- Forked files grow roughly linearly because the forked rollout physically copies prior transcript history before appending the new turn.
- The files are adapter-identifiable: their first `session_meta` line includes `"originator":"codex-teammate-adapter"`.

Recommendation:

- **Cleanup is not urgent or blocking.**
- If we want hygiene anyway, the lowest-risk policy is:
  - on clean shutdown, scan `~/.codex/sessions/**/rollout-*.jsonl`
  - only consider files whose **mtime is older than 7 days**
  - only delete files whose first `session_meta` line parses and has `payload.originator == "codex-teammate-adapter"`
  - skip unreadable / unparsable files

That gives a concrete bound without touching normal user-owned Codex sessions.

## 2) Compact-then-fork semantics

Probe: `/tmp/v73_compact_probe.py`

Flow:

1. Start thread A.
2. Plant 3 turns with unique tokens: `ALPHA-RED-111`, `BETA-BLUE-222`, `GAMMA-GREEN-333`.
3. Call `thread/compact/start(A)`.
4. Poll `thread/read(includeTurns=True)` until a `contextCompaction` item appears.
5. Call `thread/fork(A)` to create B.
6. Read B before its first turn, then ask B to list the 3 tokens.

Results:

- A before compact: item types were just the raw turns:
  - `userMessage, agentMessage, userMessage, agentMessage, userMessage, agentMessage`
- A after compact:
  - same six items **plus** `contextCompaction`
- B immediately after fork, before any new turn:
  - again the same six items **plus** `contextCompaction`
- B could still answer correctly:
  - `ALPHA-RED-111,BETA-BLUE-222,GAMMA-GREEN-333`
- B’s rollout JSONL also physically contained:
  - the full raw A transcript
  - the compacted event (`"type":"compacted"`)

Empirical conclusion:

- **B inherits A’s post-compact state** in the sense that `thread/read(..., includeTurns=True)` on B already includes the `contextCompaction` item.
- But **Codex does not prune raw history from storage** when it forks. The forked rollout still copies the full earlier transcript onto disk, with the compacted event appended.

So the most accurate answer is: **compact-then-fork yields a mixed state** — B is not a summary-only branch; it inherits the compaction marker/state, but the raw pre-compact transcript is still present in the stored rollout.

## 3) Fork-after-turn materialization race

Current code reality:

- `_start_or_fork_thread` does **one** `is_thread_materialized(resume_thread_id)` probe (`src/codex_teammate/codex.py:573-588`)
- if false, it falls back immediately to fresh `thread/start`
- there is **no retry loop** today
- note: `app_server_invoke`’s docstring says “within a short retry window,” but the implementation currently has only a single probe + fallback

I checked the race two ways.

### Same-client immediate check

Probe: `/tmp/v73_immediate_post_completion_probe.py`

Immediately after receiving the matching `turn/completed` notification for a fresh non-ephemeral thread:

- rollout path already existed: **true**
- `thread_read(includeTurns=True)` succeeded: **true**
- turn count was already materialized: **1**

### Cross-process “next task” check

Probe: `/tmp/v73_materialization_probe.py`

For 5 trials:

1. client 1 started a fresh thread and ran one tiny turn
2. on `turn/completed`, client 1 closed
3. client 2 immediately started and attempted `thread_read(includeTurns=True)` and `thread_fork`

Results: **5/5 success**

- `path_exists_immediate`: true in all 5 trials
- `thread_read(includeTurns=True)`: true in all 5 trials
- `thread_fork`: true in all 5 trials
- measured probe latency (client 2 startup + init + check): **~773-826 ms**

Conclusion:

- I could **not** reproduce a materialization race on `codex-cli 0.122.0`.
- By the time a realistic “next task” starts, the rollout file was already there in every trial.
- This is **not a blocker**.

Minimal hardening, if we still want belt-and-suspenders:

- only on the `resume_thread_id != None` path
- if the first `is_thread_materialized()` probe returns false, poll every **100 ms for up to 1 s** (or `250 ms x 4`) before falling back
- keep the existing fallback after that

That would match the current docstring, costs almost nothing, and would cover rare filesystem lag without changing steady-state behavior.

## 4) Fork fallback behavior

Fallback path trace (`src/codex_teammate/codex.py:573-600`):

1. `is_thread_materialized(resume_thread_id)` is called.
2. If it raises, code logs `app_server.fork_materialization_probe_failed` and sets `materialized = False`.
3. If `materialized` is false, code logs `app_server.fork_fallback_unmaterialized`.
4. It then returns `client.thread_start(**thread_kwargs)`.
5. No explicit signal is surfaced to the caller beyond logs.

What happens after that:

- `app_server_invoke` returns the fresh thread id.
- `loop.py:473-480` always stores that latest thread id into `state.app_server_last_thread_id`.
- So once fallback happens, the lineage silently rebases onto the new blank thread and future tasks continue from there.

Answer:

- **Yes, it falls back cleanly** (no crash, no protocol error) by switching to a fresh `thread/start`.
- **Yes, lineage is silently lost** apart from the warning log.

Is that desirable?

- As a resilience mechanism: **yes** — it keeps task execution moving.
- As a UX for “cross-task memory”: **not really** — silent memory loss is surprising.

Recommendation:

- Don’t block v7.3 on this.
- But if `app_server.fork_fallback_unmaterialized` ever fires, we should probably surface it above pure logs (task summary warning, visible event, or explicit team-lead note), because otherwise the feature degrades silently.

## Bottom line

- Disk cost is tiny: about **22 KB/task** in the 5-task chain probe.
- Compact-then-fork does **not** produce a summary-only child: B inherits the compaction marker/state, while the stored rollout still contains full raw history.
- I found **no empirical materialization race** after `turn/completed`; current lack of retry is therefore not a blocker, though a tiny poll loop would make the code match its docstring.
- The real sharp edge today is **silent lineage loss on fallback**: operationally safe, but probably worth surfacing if it ever happens.
