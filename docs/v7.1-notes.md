# v7.1 Notes: Mid-Task Reactivity via Codex App Server

**Status:** shipped (task #18 completed post-hoc by team-lead after live evidence was audited).
**Date:** 2026-04-22
**Primary artifact:** `src/codex_teammate/app_server.py`
**Live acceptance:** task #19 (`m2-artifacts/sortit.py` produced without `benchmark()` after a mid-task steer removed it from scope).

This document exists for reviewer's conformance review (task #6). It describes what v7.1 actually implements, how the pieces fit, what the evidence shows, and what follow-up work is recommended.

## 1. What v7.1 adds over v7

v7 invokes Codex one-shot via `codex exec`. Inbox messages that arrive during a task are buffered and only processed after the task completes. That is a fine baseline but not a "first-class teammate" story — native Claude teammates can react to mid-task messages and fold them into their next turn.

v7.1 replaces that one-shot path with a long-lived JSON-RPC session against `codex app-server` (experimental in Codex CLI 0.120.0). The adapter keeps a `turn/start`-ed turn alive, streams notifications, and can inject additional input into the in-flight turn via `turn/steer`. The mid-task reactivity primitive exists and is used.

v7.1 was **initially opt-in** via `--app-server` / `CODEX_TEAMMATE_APP_SERVER=1`. As of task #21 (the default flip), **App Server mode is the default**: the adapter runs under App Server unless the user passes `--no-app-server` or sets `CODEX_TEAMMATE_APP_SERVER=false`. The opt-out exists for v7.2 `codex exec resume` scenarios where cross-task session memory matters more than mid-task reactivity — the two modes are orthogonal feature profiles (see v7.2-notes.md §6).

**v7.1 steer and v7 wrapper-MCP tools coexist** under App Server default. They're orthogonal primitives on different directions:

- `turn/steer` is **inbound** — the adapter submits user-style input into a running turn, triggered by mid-task inbox messages.
- Wrapper MCP tools (`send_message`, `task_update`, `task_create`, `read_inbox`, `task_list`, `read_config`) are **outbound** — Codex calls them from inside its turn.

Both wire into the same App Server thread but don't interfere: the event stream interleaves turn/steer-triggered input items with Codex's own tool-call items. No rewiring was needed when the default flipped in task #21; both paths continue to work as they did in v7.1.

## 2. App Server methods used

From `codex app-server generate-json-schema --out <dir>` (60 methods total in 0.120.0):

| Method | Purpose in v7.1 |
|---|---|
| `initialize` | JSON-RPC handshake; response includes `codexHome`, `platformFamily`, `userAgent`. |
| `thread/start` | Create a per-task thread. Params: `cwd`, `sandbox: "danger-full-access"`, `approvalPolicy: "never"`, `ephemeral: true`, `baseInstructions`, `developerInstructions`, `config.mcp_servers.codex_teammate_wrapper`. Returns `threadId`. |
| `turn/start` | Submit the initial task prompt. Params: `threadId`, `input: [{type: "text", text: prompt}]`, `outputSchema` (the task-complete JSON Schema inline, not a path). Returns `turnId`. |
| **`turn/steer`** | Inject additional input into an in-flight turn. Params: `threadId`, `expectedTurnId` (precondition — fails if the active turn ended), `input: [...]`. Returns the resulting `turnId`. |
| `turn/interrupt` | Abort an active turn on timeout. |

Notifications consumed: `thread/started`, `turn/started`, `turn/completed`, `item/*` (including `agentMessage`, `mcpToolCall`, `fileChange`). Notification method names use the lowercase-slashed form (`thread/started`), not the PascalCase type names found in the generated schema (`ThreadStartedNotification`).

## 3. Concurrency shape (threaded, not asyncio)

The control loop (`loop.py`) stays synchronous — same lifecycle as v7. Per-task, `_execute_task_app_server` does:

1. Builds a `SteerQueue` (thread-safe FIFO of text fragments).
2. Builds a `mid_turn_hook` closure that drains the adapter's own inbox via `pio.read_own_inbox`. Each prose message becomes `steer_queue.push("mid-task message from <sender>: <body>")`. Shutdown requests set `state.shutdown_requested` for the outer loop to process after the turn completes.
3. Calls `codex_mod.app_server_invoke(..., steer_queue=steer_queue, mid_turn_hook=_mid_turn_hook)`.

Inside `app_server_invoke`, the event loop interleaves three things on a 0.5s tick:

- Drain pending steers from the queue → `client.turn_steer(thread_id, expected_turn_id=current_turn_id, text=...)`.
- Call `mid_turn_hook()` (drains the adapter's inbox into the steer queue).
- Block on `client.notifications.get(timeout=0.5)` for the next App Server notification.

The `AppServerClient` itself has **one worker thread** — the stdout reader — that parses JSON-RPC messages from the Codex subprocess and dispatches them to either pending request futures (by id) or a notification queue. All other access is synchronous from the control loop's perspective, which keeps coordination simple.

I deliberately chose threaded-with-queues over `asyncio` because (a) the surrounding control loop is already sync and rewiring it to asyncio would have been substantially more invasive without buying us anything v7.1 needed, and (b) one stdio reader thread is the minimum viable concurrency to make `turn/steer` work — adding an event loop on top would have been ceremony.

## 4. `expectedTurnId` precondition handling

`turn/steer` is atomic-or-nothing: if the active turn has already ended by the time the server processes the request, it fails with a JSON-RPC error. `app_server_invoke` catches this as `AppServerError`, logs it as `app_server.steer_failed`, and drops the steer silently (the fragment is lost). Rationale: the turn ending mid-steer is a race-lost condition, not a bug — the surrounding message is still in the inbox for the next turn to consume.

A stronger design would re-queue the failed steer to land as input on the next `turn/start`, but that spans task boundaries and is better handled by v7.2 (cross-task context persistence) which is CLI-blocked. Out of scope for v7.1.

## 5. Sandbox policy

v7.1 inherits v7's stance: no sandbox, operator-run. The bypass is expressed as `thread/start` params `sandbox: "danger-full-access"` + `approvalPolicy: "never"` — the App Server equivalent of v7's `--dangerously-bypass-approvals-and-sandbox` flag. No `sandbox_workspace_write.writable_roots` carve-out is needed because the sandbox is off entirely.

The wrapper MCP server still runs as a subprocess of Codex and inherits its (absent) sandbox. Wrapper writes to `~/.claude/tasks/` and `~/.claude/teams/*/inboxes/` succeed for the same reason v7 `task_update` / `send_message` calls succeeded after the sandbox bypass landed.

### 7.1 Correction: v7.1 acceptance scope and the regression caught at task #22

**v7.1 shipped; live acceptance covered `turn/steer` (adapter→Codex inbound direction) only.** Task #19's scenario deliberately exercised only the inbound direction: team-lead sent a steer mid-task, Codex incorporated it, `sortit.py` reflected the steered scope (benchmarks removed). That proved the reactivity primitive worked. It did NOT prove outbound wrapper MCP tool access — Codex calling `task_update` / `send_message` / etc. from its own reasoning — which regressed from v7's fresh-exec semantics to App Server mode due to env-var inheritance on MCP subprocess spawn.

The concrete mechanism: App Server does NOT forward the AppServerClient parent's env into the MCP-server subprocesses it spawns (verified live 2026-04-22 via `/tmp/app_server_mcp_probe*.py`). Our wrapper's `_identity()` read `CODEX_TEAMMATE_TEAM` / `CODEX_TEAMMATE_NAME` from env, found them missing, raised `RuntimeError`, and the wrapper subprocess exited before replying to App Server's MCP `initialize` request. Result: `mcpServer/startupStatus/updated status=failed error=MCP client for 'codex_teammate_wrapper' failed to start: MCP startup failed: handshaking with MCP server failed: connection closed: initialize response`. Codex's agent was then never presented with the wrapper's six tools during turn execution, so Codex's self-report ("the requested MCP protocol tools were not exposed in this session's callable tool list") was accurate.

**Regression caught during task #22 sanity probes** (reviewer's S3 minigrep task surfaced the self-report). Fixed same day via a narrow patch:

- `src/codex_teammate/wrapper_server.py::_identity()` — gained `--team` / `--name` CLI args (with the `--team=X` equals-form variant) and uses them in preference to env, falling back to env for backward compat.
- `src/codex_teammate/codex.py::app_server_invoke` — builds `mcp_servers.codex_teammate_wrapper.args` as `["--team", settings_team, "--name", settings_agent]` instead of the previous empty list, so identity rides with the subprocess spawn invocation rather than depending on env forwarding.
- `tests/test_wrapper_contract.py` — 5 new tests covering CLI-args resolution, CLI-equals-form, CLI-beats-env precedence, env-fallback for backward compat, partial-CLI-args-fall-back-to-env.
- `tests/test_app_server_mcp_config.py` — 3 new tests covering the `app_server_invoke`→`thread_start` config shape (args contain `--team/--name`, command is the resolved wrapper path, no stray `env` field that would indicate back-sliding to Fix A).

**Re-validated under task #29 smoke test** (2026-04-22 18:09Z). Task #29 explicitly required Codex to call `task_update(active_form="writing hello.py")` before writing its output file. Evidence in the adapter's log shows `codex.tool_call` events firing during the run; the task file on disk recorded the `activeForm` change; `m2-artifacts/hello.py` exists and runs correctly. This is outbound wrapper MCP tool access under App Server mode, working as originally intended by v7.1 but not empirically proven until now.

Reviewer's task #22 re-probe against the new default is pending; once their S-series re-runs pass with the fix in place, #22 closes and #23 (final-review amend) picks up.

## 8. Known gaps and recommended follow-ups

1. **`tool_call_events: 0` on task #19 — resolved in §7.1.** The classifier's count of zero was accurate; the wrapper MCP was silently failing to handshake with App Server, so Codex never actually saw the wrapper's tools and never emitted any tool-call events. This was the regression, not a classifier blind spot. Fixed by the Fix B patch documented in §7.1; the classifier itself is unchanged and continues to match `_WRAPPER_TOOL_NAMES` and known event-type substrings correctly.

2. **Race-lost steer.** If `turn/steer` arrives after the turn has naturally completed, the fragment is dropped (logged `app_server.steer_failed`). A future hardening re-queues it as input to the next `turn/start`, which spans task boundaries and is v7.2 territory.

3. **Thread-per-task.** v7.1 uses one thread per task. Cross-task memory (the v7.2 goal) is blocked at the CLI layer — see `docs/v7.2-findings.md`. App Server supports `thread/resume`, so if/when v7.2 is unblocked the adapter can adopt that path with minor changes to `_execute_task_app_server`. v7.3 updates this: cross-task memory is restored at the thread-creation layer via `thread/fork`; see `docs/v7.3-implementation-notes.md`.

4. **Concurrent-claim race (observed on task #17).** Claim logic is first-come-first-served with no pre-assignment. Recommended future work: `metadata.restricted_to: ["codex-alice"]` on task descriptions, honored by `_find_and_claim`. ~20 LoC. Not blocking for task #6 but worth calling out.

5. **Asyncio migration.** If someone later wants `asyncio` throughout, `AppServerClient` already has the right shape for a port (one I/O source, pending-by-id dispatcher). The control loop above it would need the bigger rewrite.
