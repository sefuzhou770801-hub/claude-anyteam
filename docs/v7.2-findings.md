# v7.2 Findings: `codex exec resume` on codex-cli 0.120.0

Date verified: 2026-04-22

## Result

Blocked. `codex exec resume` is not usable for the v7.2 adapter change on the locally installed `codex-cli 0.120.0`.

## What was verified

1. `codex --version` reports `codex-cli 0.120.0`.
2. `codex exec --help` advertises the fresh-session surface the adapter currently depends on, including:
   - `--json`
   - `--output-schema`
   - `--output-last-message`
   - `-C, --cd`
3. `codex exec resume --help` advertises a narrower surface. It includes:
   - `--json`
   - `--output-last-message`
   - `--skip-git-repo-check`
   - no `--output-schema`
   - no `-C, --cd`
4. Parser check:
   - Command run:
     - `codex exec resume 00000000-0000-0000-0000-000000000000 --output-schema <tmp> --json -o <tmp.out> 'hi'`
   - Result:
     - exits with `error: unexpected argument '--output-schema' found`

## Why this blocks v7.2

The adapter's task contract is schema-constrained: every Codex run is expected to produce a final response validated by `--output-schema`, and the adapter deliberately does not parse free-form prose as a fallback. Since `codex exec resume` in 0.120.0 rejects `--output-schema`, resuming a prior session would require changing the adapter to a different, prompt-only result-parsing mechanism. The task explicitly forbids backporting resume semantics by another mechanism.

The missing `-C/--cd` flag is a secondary incompatibility. It may be possible to compensate with subprocess working-directory control, but that does not solve the missing schema surface, which is the hard blocker.

## Recommendation

Do not implement v7.2 against `codex-cli 0.120.0`. Revisit once `codex exec resume` supports the same structured-output surface the adapter requires, or once the project explicitly approves a different contract for resumed runs.
