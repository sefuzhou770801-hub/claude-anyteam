# 05 — claude_teams substrate and claude-anyteam protocol map

## Executive summary

- The durable substrate is a small file protocol: `config.json` for membership, per-agent inbox JSON arrays for messages, and per-task JSON files for task state. The project architecture calls this the core contract, with team state in `~/.claude/teams/{team}/config.json`, inboxes in `~/.claude/teams/{team}/inboxes/{name}.json`, and task work mediated by mailbox polling, task claims, idle notifications, and shutdown requests (`docs/architecture.md:9-13`).
- The vendored `claude_teams` server is the broad, host-facing MCP control plane: it can create/delete teams, spawn/kill/process teammates, mutate tasks, read inboxes/config, and scrape tmux panes (`src/claude_teams/server.py:264-295`, `src/claude_teams/server.py:298-339`, `src/claude_teams/server.py:350-536`, `src/claude_teams/server.py:539-653`, `src/claude_teams/server.py:656-780`).
- The claude-anyteam wrapper intentionally narrows that plane for routed backends. It exposes coordination tools plus shadow host tools, while blocking destructive lifecycle tools (`src/claude_anyteam/wrapper_server.py:1-20`, `src/claude_anyteam/wrapper_server.py:52-81`). This is already drift from `docs/architecture.md`, which still describes only the six coordination tools (`docs/architecture.md:108`).
- The adapter layer is not just an invocation wrapper; it is the protocol participant. It self-registers, polls only its own inbox, claims pending tasks with compare-and-set semantics, sends idle/task/block/permission messages, and chooses backend-specific turn behavior (`src/claude_anyteam/registration.py:96-187`, `src/claude_anyteam/protocol_io.py:59-130`, `src/claude_anyteam/protocol_io.py:180-268`, `src/claude_anyteam/loop.py:123-172`).
- B9's visibility envelope proposal is still mostly not in the substrate. Current code has some partial repairs — Codex App Server now counts host `commandExecution`/`fileChange`/`webSearch` as observable tool events and has a local soft non-progress steer (`src/claude_anyteam/codex.py:74-88`, `src/claude_anyteam/codex.py:742-835`) — but there is still no versioned event envelope model, no append-only `events/<agent>.jsonl`, and no wrapper-level event fan-out matching B9 §6 (`bug-triage/B9-visibility-parity-investigation.md:590-632`, `bug-triage/B9-visibility-parity-investigation.md:865-899`).

## Protocol invariants enforced only by code

### Names, membership, and config

- Team names must match `^[A-Za-z0-9_-]+$` and be at most 64 chars; this is enforced in `create_team()`, not in `TeamConfig` (`src/claude_teams/teams.py:24-50`, `src/claude_teams/models.py:65-74`).
- Agent names must match the same regex, be at most 64 chars, and cannot be `team-lead`; this is enforced in the spawner, not in `TeammateMember` (`src/claude_teams/spawner.py:99-106`, `src/claude_teams/models.py:30-45`).
- The member union discriminator is not a stored explicit tag; it is inferred from whether the raw member object has a `prompt` key (`src/claude_teams/models.py:48-62`). Recommendation: add an explicit `memberType` or `role` while preserving the discriminator for backward compatibility.
- Team deletion is refused while any `TeammateMember` remains, but that invariant is code-only in `delete_team()` (`src/claude_teams/teams.py:138-146`). Recommendation: expose lifecycle state in config (`active|shutting_down|deleted`) and require a graceful drain path before deletion.
- Removing `team-lead` is forbidden in `remove_member()`, not by config schema (`src/claude_teams/teams.py:167-172`). Recommendation: represent the lead as a separate top-level field or validate `members` with a required singleton lead role.
- Anyteam registration is idempotent by name and only self-heals `agentType` and `backendType` on an existing row; all other stale fields remain unchanged (`src/claude_anyteam/registration.py:120-139`). Recommendation: include a schema-versioned `adapter` object and update it atomically rather than relying on selected top-level fields.
- Anyteam registration writes `tmuxPaneId="in-process"` even though routed peers are launched through the host pane path for TUI presence; architecture explicitly says the TUI row comes from host in-memory spawn state and config registration is only the companion piece (`src/claude_anyteam/registration.py:141-159`, `docs/architecture.md:85-96`). Recommendation: separate durable `transport`/`visibility` metadata from `tmuxPaneId` so an adapter is not forced to impersonate a pane id.

### Message routing and inbox semantics

- Server direct `send_message(type="message")` requires non-empty content, summary, and recipient; validates sender and recipient membership; forbids self-send; and forbids teammate-to-teammate direct messages unless the recipient is `team-lead` (`src/claude_teams/server.py:379-397`). Recommendation: codify allowed routing policies in a message schema/policy table rather than hidden branching.
- Server broadcast is lead-only (`src/claude_teams/server.py:423-445`). Wrapper broadcast uses `to="*"` and is available to routed teammates (`src/claude_anyteam/wrapper_server.py:206-221`). Recommendation: decide whether peer broadcast is intentionally allowed; if yes, rename/document it as `broadcast_message`, and if no, align wrapper policy with server policy.
- Server shutdown requests cannot target `team-lead` and require the recipient to be a member (`src/claude_teams/server.py:447-458`). Wrapper does not expose shutdown request/response at all (`src/claude_anyteam/wrapper_server.py:178-243`). Recommendation: keep lifecycle in the adapter, but expose a read-only `request_shutdown` status if models need to explain why they are stopping.
- `send_message()` appends a `<system_reminder>` to direct/broadcast message content in the vendored server path, but wrapper `send_message()` writes the body exactly as given (`src/claude_teams/server.py:253-261`, `src/claude_teams/server.py:403-410`, `src/claude_anyteam/wrapper_server.py:235-243`). Recommendation: move sender/reminder metadata into structured fields, not text mutation.
- `InboxMessage` has no `extra="allow"`, and read/mark paths deserialize into `InboxMessage` then rewrite with `model_dump()`, which strips unknown fields (`src/claude_teams/models.py:90-99`, `src/claude_teams/messaging.py:53-70`). The adapter knows this and asserts it only marks its own inbox read (`src/claude_anyteam/protocol_io.py:59-75`). Recommendation: preserve raw entries and update only the `read` bit by index/message id.
- `parse_protocol_text()` treats unrecognized or malformed JSON as prose and never raises (`src/claude_anyteam/messages.py:156-193`). Recommendation: add an explicit `protocol_error` diagnostic for malformed known `type` values so corrupt control messages do not silently become chat.
- A literal text prefix `steer:` is promoted to `SteerIn` even when it is not JSON (`src/claude_anyteam/messages.py:164-166`). Recommendation: keep it as compatibility, but mark the parsed steer with `source="legacy_text_prefix"` for auditability.
- Shutdown request wire format is ambiguous enough that the inbound model accepts `requestId`/`request_id` by alias/populate behavior (`src/claude_anyteam/messages.py:39-55`). Recommendation: emit only one canonical key plus `schema_version`, and accept legacy aliases only at the parser edge.
- Permission response matching scans only unread messages from `team-lead`, requires JSON `type="permission_response"`, validates `request_id`, and then marks exactly that entry read (`src/claude_anyteam/protocol_io.py:271-313`). Recommendation: give every inbox message an id so matching/ack is independent of list position and JSON rewrites.

### Task state and claiming

- Task ids are numeric filenames allocated as `max(existing)+1`; non-numeric JSON files are ignored (`src/claude_teams/tasks.py:52-60`). Recommendation: keep numeric display ids but add immutable UUIDs to avoid reuse/ambiguity after deletion.
- Task subjects must be non-empty and the team must exist; this is enforced only in `create_task()` (`src/claude_teams/tasks.py:71-74`). Recommendation: encode this in JSON Schema for task-create inputs and TaskFile validation.
- Task claim is a custom compare-and-set helper that refuses non-pending tasks and tasks owned by someone else; it writes under the task lock and deliberately does not call `tasks.update_task()` to avoid same-process deadlock on separate `FileLock` instances (`src/claude_anyteam/protocol_io.py:86-130`). Recommendation: make CAS a first-class substrate operation (`claim_task(expected_status, expected_owner, new_owner, active_form)`) and stop duplicating task-write logic outside `tasks.py`.
- Task status is monotonic for normal statuses: `pending -> in_progress -> completed`; transitions backward are rejected (`src/claude_teams/tasks.py:19-20`, `src/claude_teams/tasks.py:161-169`). `deleted` is a special side path (`src/claude_teams/tasks.py:253-284`). Recommendation: model status transitions explicitly and add a `blocked` terminal/nonterminal status instead of encoding blocked state in activeForm/metadata.
- A task cannot block or be blocked by itself, dependencies must reference existing task files, and cycles are rejected with `_would_create_cycle()` (`src/claude_teams/tasks.py:27-49`, `src/claude_teams/tasks.py:129-160`). Recommendation: store dependency updates as a transaction object with both forward/reverse edges and validate it under the task lock.
- A task cannot be set `in_progress` or `completed` while any effective blocker exists and is not completed (`src/claude_teams/tasks.py:170-183`). The adapter's `_blocked()` also treats blockers with status `deleted` as non-blocking, but `tasks.update_task()` does not include `deleted` in that blocker-complete check (`src/claude_anyteam/loop.py:564-574`, `src/claude_teams/tasks.py:170-183`). Recommendation: define blocker resolution semantics once (`completed|deleted` or only `completed`) and share it.
- Completing a task removes that task id from other tasks' `blockedBy` lists; deleting a task removes it from both `blockedBy` and `blocks` of others (`src/claude_teams/tasks.py:235-284`). Recommendation: include an audit/event entry for automatic dependency rewrites.
- Metadata patch semantics are merge-by-key and `null` deletes a key (`src/claude_teams/tasks.py:226-233`). Recommendation: reserve a namespaced subtree such as `metadata.visibility` and validate known namespaces while allowing opaque peer metadata elsewhere.
- Wrapper `task_update()` refuses updates to tasks owned by someone else, but lets a model set `owner` when the current owner is itself/empty and does not validate that the new owner is a team member (`src/claude_anyteam/wrapper_server.py:267-285`). Recommendation: delegate wrapper updates through the server validation path or perform the same owner-membership check.
- Adapter task assignment messages are intentionally informational; the shared task list is source of truth (`src/claude_anyteam/loop.py:187-190`). Recommendation: rename `task_assignment` to `task_assignment_notice` or include `task_version` so receivers can detect stale notices.

### Loop behavior and backend turn semantics

- Both Codex and Gemini poll their own inboxes, then claim/execute, then send idle at most once per 60 seconds while no claimable task exists (`src/claude_anyteam/loop.py:134-164`, `src/claude_anyteam/backends/gemini/loop.py:161-187`). Recommendation: write idle state into task/member metadata as well as mailbox so it is queryable without consuming inbox messages.
- Shutdown is idempotent by `request_id`; mid-task shutdown is rejected with in-flight task feedback but sets `shutdown_requested` so the loop exits after the task finishes (`src/claude_anyteam/loop.py:303-332`, `src/claude_anyteam/backends/gemini/loop.py:326-345`). Recommendation: add durable shutdown state (`requested_at`, `deferred_for_task`) to the member row or a lifecycle file.
- On clean approved shutdown, loops deregister; on crash or unapproved exit they intentionally leave a member row for inspection (`src/claude_anyteam/loop.py:105-120`, `src/claude_anyteam/backends/gemini/loop.py:111-119`). Recommendation: add `last_seen_at`/`exit_state` fields to distinguish zombie/crashed/working without deleting evidence.
- Codex prose replies can be delivered either as `last_message` or through wrapper `send_message`; the loop avoids a double-send fallback when `tool_call_events > 0` and no final text exists (`src/claude_anyteam/loop.py:260-267`). Gemini now has the same guard (`src/claude_anyteam/backends/gemini/loop.py:284-291`). Recommendation: make backend result include an explicit `delivered_reply=true` flag rather than inferring from tool-call count.
- Codex App Server mid-turn handling only converts prose to steer and handles shutdown; other protocol messages observed mid-turn are ignored (`src/claude_anyteam/loop.py:809-838`). Recommendation: queue non-shutdown protocol messages with a visible `deferred_until_turn_end` event.
- `SteerQueue` has no backpressure and only FIFO best-effort; steers are dropped if the turn ends before delivery (`src/claude_anyteam/codex.py:908-918`). Recommendation: store steer acknowledgements in inbox/task metadata so the sender can tell `queued`, `delivered`, `expired`, or `dropped`.
- Gemini `SteerIn` rejects non-lead senders, rejects empty messages, coerces invalid `expires_after_turns` to one, and injects only applicable steers into the next prompt prefix capped at 8192 chars (`src/claude_anyteam/backends/gemini/loop.py:211-268`). Recommendation: make `SteerIn` delivery semantics explicit per backend (`mid_turn` vs `next_turn`) in a response/event.
- Gemini ACP `trusted` mode auto-allows permission requests; `default`/`plan` modes bridge to lead inbox and block for a matching permission response (`src/claude_anyteam/backends/gemini/acp_client.py:254-321`). Codex runs with `approval_policy="never"` and `sandbox="danger-full-access"` in App Server mode (`src/claude_anyteam/codex.py:699-713`). Recommendation: expose backend trust posture to the lead in member metadata and implement permission envelopes backend-neutrally.

## Race conditions and lock semantics

| Surface | Current semantics | Race / leak | Elegant recommendation |
|---|---|---|---|
| `config.json` vendored writes | `write_config()` writes to a temp file and `os.replace()`s config (`src/claude_teams/teams.py:119-135`). `add_member()` and `remove_member()` are read-modify-write without a lock (`src/claude_teams/teams.py:158-172`). | Atomic replace prevents partial config reads, but there is no mutual exclusion between concurrent vendored add/remove/write paths and anyteam registration. Lost updates are possible. | Introduce `~/.claude/teams/<team>/config.lock` and require every config read-modify-write path to use it. | `config.json` reads | `read_config()` does unlocked `read_text()` + JSON parse + Pydantic validation (`src/claude_teams/teams.py:93-99`). | Atomic replace means readers usually see old or new whole file, but they can still race with non-atomic `create_team()` initial write (`src/claude_teams/teams.py:83-85`) and with external writers. | Make `create_team()` use the same atomic write helper and expose a tolerant read helper that retries JSON parse failures for transient external writes. |
| Inbox append/read | Append and mark-as-read use `inboxes/.lock`, but write with plain `path.write_text()` (`src/claude_teams/messaging.py:53-70`, `src/claude_teams/messaging.py:146-158`). | Permission responses | The scanner locks the teammate inbox, searches unread lead messages for a matching `permission_response`, marks only that list entry read, and rewrites the file (`src/claude_anyteam/protocol_io.py:271-313`). | The matching protocol is request-id based but message-id-less; if the lead sends malformed or duplicate decisions, only the first valid match wins and malformed known responses are only logged (`src/claude_anyteam/protocol_io.py:286-313`). | Store permission requests/responses in a keyed `requests/<id>.json` file or add inbox message ids and response dedupe semantics. |
| Task create/update | `create_task()`, `update_task()`, and `reset_owner_tasks()` use `tasks/<team>/.lock` (`src/claude_teams/tasks.py:77-90`, `src/claude_teams/tasks.py:118-123`, `src/claude_teams/tasks.py:306-325`). | Writers are serialized, but writes are non-atomic and readers (`get_task()`, `list_tasks()`) are unlocked (`src/claude_teams/tasks.py:95-101`, `src/claude_teams/tasks.py:289-303`). Loops catch list failures at a coarse level but can still miss a poll (`src/claude_anyteam/loop.py:502-509`). | Keep the task lock for write transactions, but write temp+replace and add reader retry. | Task claim | `claim_task()` is correctly lock-protected and CASes status/owner, but it hand-writes the task file to avoid deadlock (`src/claude_anyteam/protocol_io.py:86-130`). | The safest task-claim operation lives outside the vendored task module, so server/wrapper callers cannot reuse it directly. Separate implementations can drift. | Move `claim_task()` into `claude_teams.tasks` as a first-class primitive and make all adapters/wrappers use it. |
| Task block/complete race | `_mark_blocked()` defensively re-reads and skips if the task was already completed (`src/claude_anyteam/loop.py:855-875`). | This handles a known late race for blocked messages but not every stale writer: `task_update()` can still mutate activeForm/metadata after another status change if callers do not use CAS. | Add `expected_status`/`expected_owner` to task update; make block/complete terminal transitions compare-and-set. |
| Tmux status | `check_teammate()` resolves stored pane/window id and scrapes pane output on demand (`src/claude_teams/server.py:745-760`, `src/claude_teams/tmux_introspection.py:12-90`). | This is pull-based diagnostics, not live host visibility; stale `tmuxPaneId` yields errors but does not update membership. | Record pane liveness in member metadata through a periodic heartbeat or explicit status probe, and keep pane scrape as diagnostics only. |

## Concrete leak recommendations — elegant shapes

1. **Config concurrency leak:** converge on one `config.lock`, one atomic writer, and one `configVersion` CAS field for all config mutations (`src/claude_teams/teams.py:119-172`, `src/claude_anyteam/registration.py:58-69`).
2. **Inbox clobber leak:** stop array-rewrite mark-as-read from reserializing unknown fields; add message ids and `ack_messages(ids)` while preserving raw JSON entries (`src/claude_teams/messaging.py:53-70`, `src/claude_anyteam/protocol_io.py:59-75`).
3. **Task stale-write leak:** add `version`, `updatedAt`, and `expected_status/expected_owner` to task updates; make claim/block/complete all CAS operations in `claude_teams.tasks` (`src/claude_anyteam/protocol_io.py:86-130`, `src/claude_anyteam/loop.py:855-887`).
4. **Blocked-state leak:** add first-class `blocked` / `needs_attention` task status and keep `task_blocked` as an event, not the only durable signal (`src/claude_teams/models.py:82-87`, `src/claude_anyteam/protocol_io.py:201-217`).
5. **Visibility-event leak:** implement B9 `VisibilityEvent` models/schemas and `events/<agent>.jsonl`; then fan out rate-limited summaries to inbox and rolling state to `metadata.visibility` (`bug-triage/B9-visibility-parity-investigation.md:590-632`, `bug-triage/B9-visibility-parity-investigation.md:865-899`).
6. **Wrapper instrumentation leak:** wrap every `EXPOSED_TOOLS` handler with start/completed/failed visibility-event emission, including shadow host tools, with redacted previews and output-size caps (`src/claude_anyteam/wrapper_server.py:52-68`, `bug-triage/B9-visibility-parity-investigation.md:913-919`).
7. **Wrapper policy leak:** split wrapper tools into `team_tools` and `host_shadow_tools`, and document/validate each class separately. Today destructive team tools are blocked, but unrestricted shell/file/web tools are exposed without protocol-level visibility (`src/claude_anyteam/wrapper_server.py:70-81`, `src/claude_anyteam/wrapper_server.py:346-581`).
8. **Steer delivery leak:** standardize `SteerIn` with delivery status events (`queued`, `delivered_mid_turn`, `delivered_next_turn`, `expired`, `dropped`). Codex and Gemini currently have different semantics and neither sends a durable ack (`src/claude_anyteam/codex.py:908-918`, `src/claude_anyteam/backends/gemini/loop.py:211-268`).
9. **Permission asymmetry leak:** make permission request/response a backend-neutral protocol feature with trust posture in member metadata; Gemini ACP uses it, Codex bypasses approvals (`src/claude_anyteam/backends/gemini/acp_client.py:254-321`, `src/claude_anyteam/codex.py:699-713`).
10. **Task completion backend-name leak:** rename `codex_exit_code` to `backend_exit_code` and add `backend`, keeping aliases for old readers (`src/claude_anyteam/messages.py:142-154`, `src/claude_anyteam/backends/gemini/loop.py:538-542`).
11. **Tool-list documentation leak:** update `docs/architecture.md` and tests from code-generated `EXPOSED_TOOLS`; current docs say six tools while code exposes thirteen (`docs/architecture.md:108`, `src/claude_anyteam/wrapper_server.py:54-68`).
12. **Effort-surface leak:** either support `minimal` in the Codex adapter or revise the protocol docs/capability metadata so the five-tier surface is not promised where the CLI parser rejects it (`docs/architecture.md:120-137`, `src/claude_anyteam/config.py:129-134`).
13. **Spawn/visibility leak:** stop overloading `tmuxPaneId="in-process"` as a visibility hint. Add an explicit external teammate registration/status mechanism or at least a diagnostic that tells the lead when config membership exists but host in-memory mirror presence does not (`src/claude_anyteam/registration.py:141-159`, `docs/architecture.md:85-96`).
14. **Unclassified narrowing leak:** classify `task_get` in wrapper policy so every substrate tool is either exposed or intentionally blocked with rationale (`src/claude_teams/server.py:612-619`, `src/claude_anyteam/wrapper_server.py:52-81`).
15. **Malformed-protocol silence leak:** when a message looks like JSON with a known-ish protocol `type` but fails validation, send a `protocol_error` diagnostic instead of treating it as prose (`src/claude_anyteam/messages.py:156-193`).
