# v7.2 Notes: Cross-Task Context via `codex exec resume`

**Status:** shipped via Option A (per task #20). Prior "blocked-with-findings" outcome
is preserved in task-metadata history; `docs/v7.2-findings.md` remains as the
verified CLI-gap reference that motivated Option A.
**Date:** 2026-04-22
**Primary artifacts:** `src/codex_teammate/schema_validation.py`, extensions to
`src/codex_teammate/codex.py::run` and `src/codex_teammate/loop.py::_invoke_codex_for_task`.
**Live acceptance:** chained tasks #24 → #25; Codex's task-complete message on #25
said *"refactoring the **previously added** `triangle_number(n)` function"* —
first-person evidence that session memory crossed the task boundary via `exec resume`.

This document exists for reviewer's conformance review (task #23/§6). It
describes what v7.2 actually ships, how it recovers rigor on top of a
CLI surface that's narrower than v7's, what the evidence shows, and what
follow-up work is recommended.

## 1. What v7.2 adds over v7

v7 invokes `codex exec` fresh per task. Every task starts from a blank
Codex session: no memory of prior tasks, no carried context. The adapter
re-provides identity and tool descriptions in every prompt.

v7.2 makes the **second and subsequent tasks for the same adapter identity**
invoke `codex exec resume <session_id>` instead of fresh `codex exec`.
Codex resumes the prior session's memory, so it can be told "refactor the
function you just wrote" without the adapter having to re-inject what that
function was. The session id comes from the `thread.started` JSONL event
on the first fresh run, captured into `LoopState.codex_session_id` and
passed back into `codex.run` on the next invocation.

## 2. Why Option A (Python-side validation on top of `resume`)

`codex exec resume` on codex-cli 0.120.0 AND 0.122.0 does NOT accept
`--output-schema`. A 0.122.0 upgrade was attempted during task #20 (team-lead
installed it specifically to check) and the flag is still absent. The
pre-existing `docs/v7.2-findings.md` records the reproducer.

Rather than:
- **Block v7.2 again** — inconsistent with the goal of actually shipping
  cross-task memory.
- **Backport session memory via polling hacks** — explicitly forbidden per
  team-lead's escalation rule.
- **Switch to App Server for everything** — App Server gives us v7.1's
  mid-task reactivity but doesn't itself provide a resume primitive that
  matches exec-resume's semantics; conflating the two buys nothing and
  creates a coupling we don't want.

We chose **Option A: ship `codex exec resume` as-is and validate the
output in Python.** Flow:

1. Codex writes its final message to `--output-last-message <tmpfile>`
   (no schema enforcement at the CLI layer).
2. The adapter reads the file, parses as JSON, validates against
   `schemas/task-complete.schema.json` using `jsonschema`.
3. On validation failure, the adapter retries the whole `codex exec
   resume` invocation once with a **firmer prompt** (`"PRIOR ATTEMPT
   FAILED: Return ONLY the JSON object..."`) and re-validates.
4. Two failures → the task is marked `blocked` with the validation reason
   in `task_blocked`. No canned stub is ever sent.

The rigor lost at the CLI (schema enforcement) is recovered at the
adapter (parse + validate + retry). The retry prompt is explicit enough
to pin Codex's output shape even though `--output-schema` isn't available
on the resume flag set.

## 3. `codex exec resume` invocation

Exact argv shape built by `codex.run(..., resume_session_id=<id>)`:

```
codex exec resume <session_id>
  --json
  --skip-git-repo-check
  --dangerously-bypass-approvals-and-sandbox
  --output-last-message /tmp/codex-last-<...>.txt
  -c mcp_servers.codex_teammate_wrapper.command="..."
  -c mcp_servers.codex_teammate_wrapper.args=[]
  <prompt>
```

Compared to the fresh `codex exec` path (v7), the resume path **drops**:

- `--output-schema <path>` — not accepted by `resume` (CLI gap).
- `-C <cwd>` — not accepted by `resume`. Cwd is inherited from the
  adapter's own cwd at invocation time. If a future task needs a
  different cwd, it must be communicated via the prompt content itself;
  the adapter does not re-enter a `-C` override on resume.

Everything else carries over: `--dangerously-bypass-approvals-and-sandbox`
for the sandbox bypass (same operator-trust-envelope rationale as v7),
`--output-last-message` for the response sink, the MCP-wrapper `-c`
overrides, and `--json` for the JSONL event stream (which is also where
we capture the session id on the fresh path).

Regression tests for this exact shape live in
`tests/test_codex_invocation_shape.py` under the `test_resume_argv_*`
names.

## 4. Python-side schema validation

Module: `src/codex_teammate/schema_validation.py`. Two functions:

- `load_schema(path)` — reads a JSON Schema file.
- `parse_and_validate(text, schema) -> (parsed, err)` — parses the
  last-message text as JSON, validates against the schema, returns
  `(dict, None)` on success or `(None, reason)` on any failure. Never
  raises; the caller has its own decision logic.
- `inline_schema_prompt_fragment(schema)` — renders the strict-prompt
  preamble used on the resume path. Lives in one place so the retry
  prompt reuses the same language.

Deliberately firm contracts:

- Markdown fences (```json\n... \n```) are NOT auto-stripped. If Codex
  wraps the JSON, we surface it as a prompt-discipline failure on the
  first attempt and the retry prompt gets stricter. The test
  `test_markdown_fences_are_NOT_unwrapped` guards this.
- JSON arrays and scalars are rejected even though the JSON itself parses
  — the task-complete schema expects an object.
- `additionalProperties: false` in the schema means Codex inventing
  extra fields fails validation, not silent acceptance.

## 5. Session-ID capture and per-identity tracking

`codex exec --json` emits `{"type":"thread.started","thread_id":"<uuid>"}`
as the first event. The adapter captures that into
`CodexResult.session_id`, which `_execute_task` stores in
`LoopState.codex_session_id`.

Scope: **in-memory, adapter-lifetime only.** If the adapter restarts,
the session id is gone and the next task will be a fresh `exec`.
Persistence across adapter restarts is explicitly out of v7.2 scope;
it's the next iteration of the feature and requires a persistence store
plus a strategy for reconciling stale sessions (the original task #17
brief hinted at this as a future concern).

One session id per adapter identity. The adapter is single-identity at
runtime, so in practice this is one slot. The structure is kept as an
attribute on `LoopState` rather than a dict so it composes cleanly with
shutdown/restart semantics.

## 6. Dispatch: which path does `_execute_task` take?

`loop.py::_invoke_codex_for_task` is the single dispatch function.
Branch order (first match wins):

1. `settings.app_server` → **v7.1/v7.3 App Server path.** In the original
   v7.1/v7.2 framing this meant thread-per-task with no cross-task context;
   as of v7.3 the same branch also supports cross-task memory via
   `thread/fork`, while still providing mid-task reactivity via
   `turn/steer`. See `v7.1-notes.md` §3 and
   `v7.3-implementation-notes.md`.
2. `state.codex_session_id is not None` → **v7.2 resume path.** No
   `--output-schema`, Python-side validation, retry-once-then-block.
3. Otherwise → **v7 fresh-exec path.** CLI-enforced `--output-schema`.

Historically (v7.1 + v7.2), path 1 and path 2 were mutually exclusive:
under App Server, v7.2 resume did not apply, so users effectively chose
between "reactivity" and "session memory." As of v7.3 that framing is
superseded: App Server mode also carries cross-task memory, just via
`thread/fork` rather than `codex exec resume`. The branches remain
mutually exclusive as implementation paths, but no longer as user-facing
capability profiles. `--no-app-server` still selects the v7.2
fresh-exec/resume implementation.

- `tests/test_schema_validation.py` (12 tests). Covers: valid parse,
  empty/whitespace rejection, non-JSON head-citation in error,
  non-object JSON rejection, missing-required-field, empty-summary
  (minLength), `additionalProperties: false` enforcement, trailing-
  whitespace tolerance, markdown-fences-NOT-unwrapped, prompt-

- `tests/test_codex_invocation_shape.py` — six new tests (`test_resume_*`
  and `test_fresh_exec_still_includes_schema_and_cwd`). Covers the exact
  argv shape on the resume path, including the absences (`--output-schema`,
  `-C`, `--cd`) and the preserved core flags. Regression guards against
  silently reverting the fresh-exec flag set.
- `tests/test_resume_dispatch.py` — six tests of `_invoke_codex_for_task`.
  Covers: first-task-is-fresh-exec, resume-path-with-validation,
  retry-once-then-succeed, two-failures-mark-blocked, nonzero-exit-on-
  retry-returns-result-for-caller (doesn't double-block), session-id
  capture forwarded to the next task.

All via mocked `subprocess.run` / mocked `codex.run`. No Codex tokens
burned in the unit path.

Ran under `codex-alice` (no `--app-server`, so resume path is exercised)
against codex-cli 0.122.0. Log extract:

```
16:34:52  task.claimed        task_id=24  subject="v7.2 live A: ..."
16:34:52  codex.invoke        schema=.../task-complete.schema.json  mcp_wrapper=true
16:35:36  codex.session_captured  session_id=019db60b-6d74-7e70-bcc1-90ca321edd19
16:35:36  codex.done          exit_code=0  structured=true  resumed=false
16:35:36  task.session_captured   session_id=019db60b-6d74-7e70-bcc1-90ca321edd19
16:35:36  task.completed      task_id=24  files=1
16:35:36  task.claimed        task_id=25  subject="v7.2 live B: ..."
16:35:36  codex.invoke        schema=null  mcp_wrapper=true
16:36:12  codex.done          exit_code=0  structured=false  session_id=<same>  resumed=true
16:36:12  task.completed      task_id=25  files=1
```

Two observable v7.2 behaviors in this trace:

1. **`resumed=false` on task #24, `resumed=true` on task #25** — dispatch
   correctly routed the first task through fresh-exec-with-schema and
   the second through resume-with-Python-validation. The `schema=null`
   on task #25's `codex.invoke` is deliberate: the resume path doesn't
   pass `--output-schema`.
2. **Task #25 completed on the first validation attempt** — `resumed=true`
   but no `task.resume_schema_fail` or retry event. Codex produced
   schema-conformant output via the stricter prompt on the first try,
   so the Python validator accepted it without invoking the retry.

On disk, `m2-artifacts/mathutil.py` has `triangle_number(n)` with the
two-line docstring (including `n * (n + 1) // 2`), plus the new
`rectangular_number(n, m)` with matching ValueError guard, plus a
`main()` that dispatches by argc. CLI checks:

```
$ python3 m2-artifacts/mathutil.py 5
15
$ python3 m2-artifacts/mathutil.py 3 4
12
```

Both correct.

The load-bearing quote comes from Codex's own task_complete summary on
#25:

> "Updated `m2-artifacts/mathutil.py` by refactoring the **previously
> added** `triangle_number(n)` function without changing its external
> behavior, expanding its docstring to a two-line form..."

That phrase — "the previously added triangle_number" — is Codex
first-person referring to the prior task's work by name, without the
adapter having re-provided context in the task-B prompt. It's the
acceptance signal for v7.2: session memory actually crossed the task
boundary.

## 9. Known limitations and recommended follow-ups

1. **Adapter-lifetime-scoped session.** If the adapter is killed or
   restarts between task A and task B, the session id is lost and the
   next task is a fresh exec. A persistence store (file-backed session
   map under `~/.claude/teams/<team>/codex_sessions.json` or similar)
   is the natural next step, paired with a staleness policy for
   long-abandoned sessions. Out of scope for v7.2.
2. **No per-task "start fresh" override.** The original task brief said
   this was optional ("add only if trivially wired"). I didn't wire it
   — it's a one-line check and can be added later if a use case
   materializes.
3. **Resume + app-server = historical non-goal (superseded by v7.3).**
   Under the original v7.2 design, `--app-server` meant App Server's
   thread-per-task model and `codex exec resume` semantics were NOT
   backported into App Server threads. That historical note remains true
   for v7.2 itself. As of v7.3, though, App Server mode also supports
   cross-task memory via `thread/fork`, so the old user-facing guidance is
   deprecated. Users who stay on App Server now get continuity on the
   default path; `--no-app-server` remains the fresh-exec/resume
   implementation, not the only path to cross-task memory. See
   `docs/v7.3-implementation-notes.md`.
4. **Retry-firmer-prompt is crude.** It just prepends a hard-coded "PRIOR
   ATTEMPT FAILED" paragraph. For tasks where the schema itself is rich
   or the agent's output strayed in a specific way (wrapped in markdown,
   extra field, etc.), a more targeted retry prompt that cites the
   specific failure mode would succeed more often. The error reasons
   from `parse_and_validate` are already descriptive; weaving them into
   the retry prompt is ~5 LoC and a reasonable future hardening.
5. **Session lineage not logged in task_complete metadata.** If a
   reviewer wants to trace "which session did task #N run against," the
   adapter's structured log has `session_id` but the task_complete
   message doesn't. Small future add.
