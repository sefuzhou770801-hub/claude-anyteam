# v7.3 Investigation: Session Continuation Under App Server

**Task:** #31 (investigation-only; do NOT implement without team-lead go-ahead).
**Date:** 2026-04-22
**Codex CLI version:** 0.122.0.
**Question:** does Codex App Server support session continuation across threads,
so v7.1 (mid-task reactivity) and v7.2 (cross-task session memory) can coexist
on a single default code path rather than being mutually exclusive modes?

**TL;DR — Outcome A.** Yes. `thread/fork` is the primitive: given a prior
thread id, it creates a new thread that inherits the prior thread's
conversational history. Verified live 2026-04-22 19:03Z: a fact established
in thread A ("favorite number is 42") was correctly recalled in a forked
thread B ("42") — same 14-second test as the v7.2 brief's acceptance criterion,
adapted to App Server semantics.

## 1. Schema findings

Methods in `codex app-server`'s `ClientRequest` oneOf on 0.122.0
(full enumeration; thread-adjacent subset):

- `thread/start` — fresh thread.
- `thread/resume` — **load an existing thread from disk by `threadId`** and
  resume it. Accepts the same thread id namespace for continuation within
  the same thread (alternative: by `history`, by `path`).
- **`thread/fork`** — load an existing thread from disk by `threadId` and
  **fork it into a new thread** with a new id that inherits the prior
  conversation context. The primitive we were looking for.
- `thread/read` — return a thread's stored content. With `includeTurns=true`,
  includes full turn/item history. Useful for debugging and reflection, not
  the continuation mechanism itself.
- `thread/rollback`, `thread/archive`, `thread/unarchive`, `thread/list`,
  `thread/loaded/list`, `thread/name/set`, `thread/metadata/update`,
  `thread/compact/start`, `thread/shellCommand`, `thread/unsubscribe` —
  not continuation-related; all accept an existing `threadId`.

**`ThreadStartParams` itself has no continuation field** (no `resumeFrom`,
`parentThread`, `continueFrom`, `seedFrom`, etc.). Continuation goes
through `thread/fork` (new-thread with parent-context) or `thread/resume`
(same-thread resumption).

### ThreadForkParams shape (the new-thread-inheriting path)

From `/tmp/codex-schemas/v2/ThreadForkParams.json`:

```
required: ["threadId"]
properties: approvalPolicy, approvalsReviewer, baseInstructions,
            config, cwd, developerInstructions, ephemeral, model,
            modelProvider, serviceTier, sessionStartSource, threadId
```

Description: *"load the thread from disk by thread_id and fork it into a
new thread"*. Response (`ThreadForkResponse`) returns a full `thread`
object with a **new** id plus all the usual runtime fields (cwd,
approval policy, sandbox, etc.).

Critical implication: **App Server persists threads to disk automatically**
unless `ephemeral: true` was set at `thread/start` time. To fork later,
the parent thread must have been started non-ephemeral. Today our adapter
calls `thread_start(..., ephemeral=True)` — that would need to change
for v7.3.

## 3. Adapter-integration LoC estimate

Rough — aimed at conveying shape, not a precise fenced-off scope doc:

- **`AppServerClient`** (`src/codex_teammate/app_server.py`): add a
  `thread_fork(thread_id, **kwargs)` helper method alongside the existing
  `thread_start`. ~10 LoC including docstring. Mirrors existing helpers.
- **`codex.py::app_server_invoke`**: grow a `resume_thread_id: str | None`
  kwarg. If None → call `thread_start` as today. If set → call `thread_fork`
  with the parent thread id. ~8 LoC. Also drop `ephemeral=True` or make it
  conditional on whether the task is first-in-adapter-lifetime.
- **`LoopState`**: add `app_server_last_thread_id: str | None`. After each
  `app_server_invoke`, store the returned thread id. Pass to the next
  invocation's `resume_thread_id`. Mirrors v7.2's `codex_session_id`
  tracking. ~5 LoC in `LoopState` + 5 at the callsite.
- **Tests**: mock fork-path tests parallel to the resume-path ones in
  `tests/test_resume_dispatch.py`. ~8 new tests following the pattern:
  fork-first-vs-start-first dispatch, fork-session-id-carry-forward,
  fork-when-ephemeral-raises-appropriately. ~120 LoC.
- **Live acceptance task**: one chained-pair analogous to v7.2's
  `triangle_number → rectangular_number` task, but under App Server
  mode. Should reuse `schemas/task-complete.schema.json`; `thread/start`
  accepts `outputSchema` so we don't lose the CLI-level schema enforcement
  that v7.2 had to give up.
- **Docs**: update `v7.1-notes.md` §8 item #3 ("Thread-per-task") to
  reflect that thread/fork now bridges to cross-task memory; add
  cross-reference in `v7.2-notes.md` §6 ("Dispatch") noting that under
  v7.3 the App Server branch also supports session continuity. ~30 lines
  of prose across the two notes files.

**Total estimate: ~40 LoC of production code + ~120 LoC of tests + ~30 lines of
doc updates.** Comparable to v7.2's landing scope; substantially smaller than
v7.1's initial implementation since the App Server plumbing is already done.

**Known unknowns that could expand scope:**

- `ephemeral` semantics: our wrapper currently spawns `thread_start(..., ephemeral=True)` so we don't pollute a user's `~/.codex/sessions/` with adapter threads. Turning this off (required for fork-ability) leaks thread history into the user's Codex session store. A follow-up hardening would periodically archive or delete threads older than some horizon. Probably +20 LoC for a "cleanup old adapter threads on shutdown" path.
- Forking across adapter restarts: thread ids are persisted on disk, so in principle we could fork from a prior adapter's thread after a restart. But that crosses into the "which thread was last mine" persistence-lookup territory v7.2 also declined to tackle. Keeping fork to same-adapter-lifetime is simpler; users wanting cross-restart lineage still fall back to v7.2 (`--no-app-server`).
- Interaction with `thread/compact/start`: Codex provides a compaction operation. Whether compact-then-fork behaves correctly (carries compacted history) vs. fork-then-compact (preserves full history up to fork point) is an empirical question — probably not worth untangling unless we hit real capacity issues.

## 4. Recommendation

**Worth implementing.** The primitive exists, works, and adapter integration is small. The result closes the explicit tradeoff documented in `v7.1-notes.md` §1 / `v7.2-notes.md` §6 — v7.1's reactivity and v7.2's continuity become compatible rather than mutually exclusive, which matches what the user said they wanted ("very feasible" threshold).

Ship v7.3 as:

- App Server default (unchanged from v7.1/v7.2 close-out).
- First task per adapter identity: `thread/start` (fresh, non-ephemeral).
- Subsequent tasks: `thread/fork` with parent thread id.
- `--no-app-server` retains current semantics: fresh-exec + `codex exec resume` path, non-App-Server, unchanged.

**Deferred explicitly to post-v7.3:**

- Cross-adapter-restart lineage (out of scope per v7.2 precedent).
- Thread store garbage collection (follow-up hardening ticket).
- Compact-then-fork semantic exploration (empirical, not urgent).
