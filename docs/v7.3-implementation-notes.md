# v7.3 Implementation Notes: App Server Cross-Task Memory via `thread/fork`

**Status:** shipped.
**Date:** 2026-04-22
**Primary artifacts:** `src/codex_teammate/app_server.py`, `src/codex_teammate/codex.py`, `src/codex_teammate/loop.py`
**Primary tests:** `tests/test_fork_dispatch.py`
**Historical predecessor:** `docs/v7.3-session-continuation.md` (the investigation that proved `thread/fork` was the right primitive)

This note is the v7.3 companion to `docs/v7.1-notes.md` and
`docs/v7.2-notes.md`. v7.3 removes the big tradeoff those two versions
carried: App Server mode no longer means "mid-task reactivity but no
cross-task memory." As of v7.3, App Server mode has both.

## 1. What v7.3 adds

Before v7.3, the adapter had two distinct continuity stories:

- **v7.1 / App Server path:** `codex app-server` + `turn/steer` for
  mid-task reactivity, but one fresh thread per task.
- **v7.2 / fresh-exec path:** `codex exec resume <session_id>` for
  cross-task memory, but no mid-task steer.

v7.3 keeps both invocation families, but upgrades the App Server path so
that the **second and subsequent tasks fork from the prior App Server
thread** instead of starting from scratch.

The primitive is `thread/fork`:

- task A: `thread/start` → turn runs → adapter captures the returned thread id
- task B: `thread/fork(parent_thread_id)` → new child thread inherits A's
  conversational history → turn runs in the forked thread
- task C: `thread/fork(B)` → lineage continues

Result: App Server mode now combines:

1. **mid-task reactivity** via `turn/steer`
2. **cross-task conversational carry-forward** via `thread/fork`

That collapses the old "reactivity vs memory" tradeoff into an
implementation-detail difference rather than a user-visible capability split.

## 2. Dispatch flow under v7.3

`loop.py::_invoke_codex_for_task` still branches in the same top-level order:

1. `settings.app_server` → App Server path
2. `state.codex_session_id is not None` → v7.2 `exec resume` path
3. otherwise → fresh `codex exec`

What changed is the behavior inside path 1.

### App Server path (v7.3)

`_execute_task_app_server(...)` now passes
`resume_thread_id=state.app_server_last_thread_id` into
`codex.app_server_invoke(...)`.

Inside `app_server_invoke(...)`:

- if `resume_thread_id is None` → call `thread/start`
- if `resume_thread_id` is set and the parent thread is materialized → call
  `thread/fork(thread_id=resume_thread_id, ...)`
- if the parent thread is not materialized → log
  `app_server.fork_fallback_unmaterialized` and fall back to a fresh
  `thread/start`

After the task completes, `_execute_task(...)` stores the current thread id in
`LoopState.app_server_last_thread_id`, so the next task can fork from it.

### Fresh-exec path (unchanged from v7.2)

When App Server is disabled, the adapter still uses:

- first task: fresh `codex exec --output-schema`
- later tasks: `codex exec resume <session_id>` + Python-side validation

So v7.3 **adds** App Server continuity without removing the v7.2 fallback.

## 3. How `resume_thread_id` is tracked

The App Server lineage slot mirrors v7.2's `codex_session_id`.

`LoopState` now holds:

- `codex_session_id` — the v7.2 fresh-exec resume token
- `app_server_last_thread_id` — the most recent App Server thread id

Scope is intentionally the same as v7.2's session memory:

- **in-memory only**
- **same adapter process lifetime only**
- **not persisted across restarts**

This is deliberate. v7.3 ships the minimum state needed to continue a single
adapter's thread lineage safely, without introducing a cross-restart registry.
If the adapter restarts, the next App Server task starts fresh and builds a new
lineage from there.

## 4. Why `ephemeral=False` matters now

`thread/fork` only works when the parent thread is materialized on disk.
That means the parent cannot have been started as an ephemeral thread.

So v7.3 changes the App Server thread-creation contract:

- **v7.1:** `thread/start(..., ephemeral=True)`
- **v7.3:** `thread/start(..., ephemeral=False)` and
  `thread/fork(..., ephemeral=False)`

This is the enabling change for cross-task memory in App Server mode.
Without it, the adapter would hit Codex's "no rollout found" / "not
materialized" error when trying to fork from the prior task.

### Clearer failure mode for ephemeral parents

`AppServerClient.thread_fork()` now rewrites the upstream low-level error into a
clearer adapter-facing one: if Codex reports "no rollout found" or "not
materialized", the raised `AppServerError` explains that the parent thread is
not materialized and likely came from `ephemeral=True`, and suggests either
starting non-ephemeral or falling back to a fresh `thread/start`.

This is covered by `tests/test_fork_dispatch.py::test_fork_when_ephemeral_raises_clear_error`.

## 5. Disk-footprint regression and mitigation

The cost of `ephemeral=False` is that adapter-created App Server threads now
persist under `~/.codex/sessions/` instead of disappearing automatically.
That is a real regression versus v7.1's more disposable per-task threads.

The investigation doc (`docs/v7.3-session-continuation.md`) identified the same
tradeoff and recommended a bounded cleanup follow-up rather than blocking the
feature. The shipped v7.3 stance is:

1. **Accept the small on-disk footprint in exchange for continuity.**
   Cross-task memory in App Server mode requires materialized parent threads.
2. **Keep lineage scoped to the current adapter lifetime.**
   v7.3 does not try to recover old thread ids across restarts, which bounds
   the number of threads any one long-running adapter will actively chain.
3. **Treat thread-store garbage collection as follow-up hardening.**
   The recommended next step is an adapter-owned cleanup/archive policy for
   older persisted threads rather than reverting to ephemeral threads and
   giving up `thread/fork`.

In other words: v7.3 knowingly trades a small session-store footprint for the
new capability, and documents cleanup as the next operational hardening step.

## 6. Materialization fallback behavior

The adapter does not assume every remembered thread id is forkable forever.
Before forking, `_start_or_fork_thread(...)` asks the App Server client whether
that parent is materialized.

If the probe says "no":

- the adapter logs `app_server.fork_fallback_unmaterialized`
- the task still runs via a fresh `thread/start`
- continuity is dropped for that task, but execution is not blocked

This is intentionally conservative. A lost lineage is better than a broken
adapter loop.

## 7. Tests and regression guards

The main v7.3 unit coverage lives in `tests/test_fork_dispatch.py`:

1. first task uses `thread/start`
2. later App Server task uses `thread/fork`
3. loop-level `app_server_last_thread_id` carries forward between tasks
4. ephemeral-parent fork errors are surfaced clearly
5. `AppServerClient.thread_fork()` sends the correct JSON-RPC params
6. model/effort overrides still reach `turn/start` on the fork path

These tests intentionally mirror `tests/test_resume_dispatch.py` so the App
Server lineage branch and the v7.2 fresh-exec lineage branch read as parallel
mechanisms.

## 8. What v7.3 does **not** solve

1. **Cross-restart App Server lineage.** Thread ids are not persisted in an
   adapter-owned registry, so a restart breaks the chain.
2. **Automatic thread-store cleanup.** Persisted sessions are left in the
   user's Codex session store for now.
3. **Compaction semantics.** `thread/compact/start` exists upstream, but v7.3
   does not yet define or test compact-then-fork behavior.
4. **Identical semantics between App Server and fresh-exec modes.** They now
   share the user-facing capability of cross-task memory, but the underlying
   implementation is still different:
   - App Server: `thread/fork`
   - fresh-exec: `exec resume` + Python-side validation

## 9. User-visible takeaway

As of v7.3, the default App Server mode is no longer "reactive but forgetful."
It is reactive **and** stateful across tasks, within one adapter lifetime.
Users who stay on the default path get both mid-task steer and cross-task memory
without having to choose between them.
