# pickle-pixel-hydrateams — research digest

## What is it
HydraTeams (`hydra-proxy`, MIT; origin `https://github.com/Pickle-Pixel/HydraTeams.git`, local commit `29c6850`) is a Node/TypeScript Anthropic-Messages translation proxy that makes native Claude Code Agent Teams teammates run on OpenAI/ChatGPT-backed models. Its headline move is not to implement a new team substrate: it hydrates an existing Claude Code teammate process with a non-Claude “brain” by redirecting `ANTHROPIC_BASE_URL` and translating requests/responses at the LLM API boundary.

## Team-substrate model
Native Claude Code Agent Teams still instantiates teammates as separate Claude Code processes; HydraTeams only changes the model endpoint those processes call (`README.md:28-30`, `architecture/ARCHITECTURE.md:28-35`). Spawn/hydration is env-var based: start `hydra-proxy`, set `ANTHROPIC_BASE_URL=http://localhost:<port>` for the Claude Code process, and optionally mark the lead with `<!-- hydra:lead -->` so lead requests pass through to Anthropic while teammate requests translate to the configured target model (`README.md:100-119`, `src/proxy.ts:13-29`, `src/proxy.ts:144-187`). IPC/mailbox/task lifecycle are the native Agent Teams surfaces: `TaskCreate`, `TaskUpdate`, `SendMessage`, file-based coordination, plan approval, and graceful shutdown remain owned by Claude Code, not by the proxy (`PRINCIPLES.md:7-35`, `architecture/ARCHITECTURE.md:91-104`). Lifecycle management is therefore split: Claude Code owns teammates and protocol state; HydraTeams owns a long-running HTTP proxy and closes only its server on SIGINT/SIGTERM (`src/index.ts:29-37`). Compared with our file-based mailbox + filelock adapter, HydraTeams does not reimplement or wrap inbox/task writes and does not expose wrapper MCP tools; it piggybacks the native Claude Code team substrate by making the process believe its LLM is still Anthropic.

## §1 (harness preservation) comparison
HydraTeams preserves the **Claude Code harness** extremely well: every teammate remains a Claude Code CLI instance with native Read/Write/Edit/Bash/Glob/Grep/WebSearch, task tools, and SendMessage because the tool execution loop is unchanged (`README.md:28-31`, `VISION.md:15-22`, `architecture/ARCHITECTURE.md:39-54`). Against claude-anyteam’s §1 moat, however, it homogenizes heterogeneous teammates into one harness body: “Codex” or “Gemini” is only the model behind Claude Code, not the Codex CLI/App Server, Gemini ACP, or Kimi harness. That means native primitives such as Codex `thread/fork`/`turn/steer`, Gemini permission bridge semantics, or Kimi skills/swarm do not flow through as first-class teammate capabilities. The system is heterogeneous at the model/price/failure-mode layer, but not at the harness-capability layer.

## §2 (visibility parity) comparison
Lead visibility is strong for operations executed by the Claude Code body: tool calls, task updates, SendMessage, and standard Agent Teams lifecycle should appear like native Claude teammates because they are native Claude teammates from the harness perspective. HydraTeams translates provider SSE back into Anthropic SSE events (`message_start`, `content_block_start`, `content_block_delta`, `message_delta`, `message_stop`) and spoofs a valid Claude model name so Claude Code keeps its normal streaming/tool UI (`JOURNEY.md:78-118`, `src/translators/response.ts:51-63`, `src/translators/response.ts:119-214`). The proxy adds its own console/file request logger with lead/teammate/warmup labels, but that is outside Agent Teams visibility and uses heuristics such as tool count/message-count proximity for session identification (`src/logger.ts:62-105`). Provider errors and rate-limit retries are logged by the proxy and surfaced as API errors, but not as rich native harness-specific event envelopes (`src/proxy.ts:195-223`, `src/proxy.ts:246-272`).

## §3 (peer efficiency) comparison
**Peer-DM RTT model:** peer messages are native Claude Code `SendMessage` tool invocations from within the teammate process, so the coordination path avoids an external adapter polling loop and should be close to pure-Claude Agent Teams RTT: model emits a tool call → Claude Code executes native SendMessage → peer inbox/TUI updates. HydraTeams provides no concrete peer-DM latency metrics, but the architecture removes several costs we currently pay: no wrapper MCP discovery per message, no custom mailbox serialization path, no backend-specific peer-DM tool exposure mismatch, and no adapter process deciding after the model turn whether to synthesize a fallback. Capability discovery cost is mostly zero because the “capability set” is the fixed Claude Code toolset sent as normal Anthropic tool definitions; the tradeoff is context/tool-list overhead (docs mention 23/31 tools in debugging examples) and total loss of per-harness unique capability manifests (`JOURNEY.md:166-182`). Idle/heartbeat strategy is not documented; it presumably inherits native Agent Teams behavior, while the proxy separately detects 0-tool warmup requests (`src/logger.ts:71-79`). Concrete latency claims are absent; the only numeric timing policy is retry backoff for upstream 429s (5 retries with 1s/2s/4s/8s/10s-ish cap) and startup compatibility endpoints such as `/v1/messages/count_tokens` estimation (`JOURNEY.md:130-152`, `src/proxy.ts:115-130`, `src/proxy.ts:209-214`).

## Hydration mechanism
This is the headline pattern: **hydrate the native Claude Code teammate body with a different provider brain while leaving the body’s tools, IPC, and lifecycle intact.** The steps are:

1. Start `hydra-proxy` with a target provider/model (`--provider openai|chatgpt`, `--model ...`, optional `--passthrough lead`) and a spoofed Claude model name; the CLI prints the `ANTHROPIC_BASE_URL` to set on teammate processes (`README.md:11-24`, `src/index.ts:9-21`, `src/config.ts:31-92`).
2. Configure lead/teammate routing explicitly with a hidden `<!-- hydra:lead -->` marker in `CLAUDE.md`; the proxy inspects the Anthropic `system` field and first few messages for that marker, while also detecting teammate prompts by the native team phrase “the user interacts primarily with the team lead” (`README.md:100-119`, `JOURNEY.md:184-197`, `src/proxy.ts:13-29`, `src/proxy.ts:152-166`).
3. The Claude Code teammate sends its ordinary Anthropic `/v1/messages` request containing system prompt, full message history, and all Claude Code tool definitions. HydraTeams strips query params, handles `/v1/messages/count_tokens`, and chooses passthrough vs translation (`architecture/ARCHITECTURE.md:91-104`, `src/proxy.ts:100-138`).
4. Request translation maps Anthropic `system/messages/tools/tool_choice` into OpenAI Chat Completions or ChatGPT Responses API shapes. Tool schemas are preserved almost 1:1 (`input_schema` → OpenAI function `parameters`), tool calls/results are converted between Anthropic content blocks and OpenAI tool/function-call records (`architecture/ARCHITECTURE.md:108-120`, `src/translators/request.ts:14-48`, `src/translators/messages.ts:15-116`, `src/translators/request-responses.ts:59-209`).
5. Response translation is a streaming state machine. For Chat Completions it tracks text block state, OpenAI tool-call indexes, Anthropic content-block indexes, and emits Anthropic SSE events; for ChatGPT Responses it maps `response.created`, `response.content_part.added`, text deltas, function-call argument deltas, and `response.completed` into the Anthropic stream Claude Code expects (`JOURNEY.md:78-118`, `src/translators/response.ts:38-230`, `src/translators/response-responses.ts:82-346`).
6. Claude Code receives a normal-looking Anthropic stream with a spoofed Claude model name, executes tool calls locally in its native harness, and feeds tool results back through the same proxy on the next LLM round (`JOURNEY.md:116-118`, `architecture/ARCHITECTURE.md:102-104`).
7. For ChatGPT subscription mode, auth is also hydrated: the proxy reads `~/.codex/auth.json`, extracts the ChatGPT account id from the JWT, and calls `chatgpt.com/backend-api/codex/responses` with Codex-style headers (`JOURNEY.md:203-256`, `src/config.ts:11-29`, `src/proxy.ts:191-235`).

The essential abstraction is a **brain/body split**. The body is Claude Code + Agent Teams + local tools + IPC. The brain is any tool-calling model whose API can be made to speak Anthropic Messages/SSE convincingly enough. That split is why peer-DM/tool visibility can stay native-fast, but it is also why unique non-Claude harness capabilities disappear.

## Lessons we should adopt
1. **Pre-hydrate control-plane context at spawn, not lazily.** HydraTeams gets native peer-DM speed because SendMessage/TaskUpdate are present inside the process from the first model round. For claude-anyteam, the analogous §3 win is to guarantee wrapper MCP protocol tools + cached capability manifests are mounted before the first task prompt, with a startup preflight that fails fast if `send_message` is missing.
2. **Use explicit routing/capability markers, not heuristics.** HydraTeams tried model name and tool-count routing, then settled on a system-prompt marker because heuristics broke under MCP/tool-count variation (`JOURNEY.md:174-197`). Our peer-steer/capability-discovery gates should similarly key off explicit manifest flags and messageKind, not prose or tool-count inference.
3. **Keep the hot coordination path in the native harness.** Their proxy is narrow: translate model API; let Claude Code execute tools and mailbox operations. Our equivalent should move fewer decisions into Python post-processing and more into native backend-visible MCP/tool surfaces, while still preserving each backend’s own harness.
4. **Implement compatibility endpoints and warmup handling as first-class startup contract.** Their bugs around query params, token counting, max-token clamps, and non-streaming warmups are exactly the sort of small API mismatches that create multi-minute stalls (`JOURNEY.md:120-152`). Add a backend startup checklist for protocol tools, token/warmup probes, and first-DM loopback before declaring a teammate available.
5. **Per-worker low-friction logs are useful, but should feed our event log.** HydraTeams’ colored console + file logs make proxy routing/debugging cheap (`src/logger.ts:111-150`); we should keep our structured visibility event log as the canonical version rather than leave these diagnostics outside the team substrate.

## Anti-patterns to avoid
1. **Do not adopt the full brain-swap architecture as our moat.** It solves §3 by flattening §1: every teammate becomes Claude Code-with-a-different-model, so Codex/Gemini/Kimi native harness primitives vanish.
2. **Avoid spoofing/masquerade as a long-term substrate contract.** Model-name spoofing and ChatGPT backend auth relay are clever but brittle; they depend on provider/private API behavior and can obscure provider-specific errors.
3. **Avoid session-identification heuristics.** Logger matching by message-count proximity and earlier tool-count routing are fragile under parallel teams and changing MCP surfaces (`src/logger.ts:81-105`, `JOURNEY.md:174-183`).

## Open questions
- Does Claude Code currently let the lead set distinct `ANTHROPIC_BASE_URL` values per spawned teammate, or does multi-proxy mode require shell/session-level hacks?
- What is the actual peer-DM RTT for GPT-backed Claude Code teammates versus native Claude teammates, especially under multi-tool loops and provider rate limits?
- How do native idle notifications, heartbeat noise, shutdown, and plan-approval events behave when the model behind the body is not Claude?
- Could a “transport avatar” pattern preserve §3 for us without flattening §1 — e.g. a native protocol sidecar that owns peer-DM/tool discovery while Codex/Gemini/Kimi still run their own harnesses?
- Which parts of ChatGPT backend subscription access are stable enough for product use, and which must remain prior-art/experimental only?
