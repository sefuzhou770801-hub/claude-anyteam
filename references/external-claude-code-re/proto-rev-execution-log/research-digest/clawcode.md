# clawcode — research digest

## What is it
Claw Code Agent is a Python reimplementation of the Claude Code agent architecture for local/OpenAI-compatible model backends; its README describes it as “Python reimplementation of the Claude Code agent architecture” and says active development lives at HarnessLab/claw-code-agent, built on instructkr/claw-code (`README.md:5-9`, `README.md:91-95`). It is public/open-source-branded (`README.md:12-18`), but this checkout has no top-level LICENSE/COPYING file and `pyproject.toml` lists authors/URLs/dependencies without a license field (`pyproject.toml:5-43`), so the precise license is unknown from the mirror.

## Architecture summary
ClawCode is a single Python agent loop (`LocalCodingAgent`) that constructs local runtimes for plugins, MCP, remote/search/account/ask/config/LSP/plan/task/team/worktree, registers plugin aliases/virtual tools, and exposes the registry as OpenAI-style function tools (`src/agent_runtime.py:102-239`, `src/agent_runtime.py:413-464`, `src/agent_tools.py:73-88`). Transport to models is flattened through a minimal OpenAI-compatible `/chat/completions` client, including streaming SSE and `tool_choice: auto` payloads (`src/openai_compat.py:134-179`, `src/openai_compat.py:250-271`). Capability exposure is local and manifest-backed: plugins can add hooks/aliases/virtual tools/blocking (`src/plugin_runtime.py:33-50`, `src/plugin_runtime.py:180-230`), while MCP discovery reads local `.claw-mcp.json`/`.mcp.json`/`.codex-mcp.json`/`mcp.json` manifests and only supports stdio server profiles (`src/mcp_runtime.py:358-386`, `src/mcp_runtime.py:441-487`). Lifecycle includes one-shot/chat/resume/background/daemon modes, session persistence, and nested child agents; child spawning creates another `LocalCodingAgent` with filtered tools and child sessions rather than launching a native external harness (`src/agent_runtime.py:2180-2212`, `src/agent_runtime.py:2214-2285`, `src/agent_runtime.py:2387-2472`, `src/agent_runtime.py:3269-3352`).

## §1 (harness preservation) comparison
ClawCode mostly **flattens** harness capabilities. Its stated goal is to reimplement the full Claude Code flow in Python, not preserve the original npm runtime or other harnesses (`README.md:91-97`, `README.md:896-900`). The model/provider boundary is a uniform OpenAI-compatible chat-completions API with function tools (`src/openai_compat.py:250-271`), and every built-in, MCP, plugin, task, team, and child-agent facility is mediated by the Python registry and prompt assembly (`src/agent_prompting.py:75-119`, `src/agent_tools.py:669-692` via README tool table). Custom agents and nested delegation are useful locally, but they are not heterogeneous harness peers: delegation filters the same Python tool registry and instantiates `LocalCodingAgent` children (`src/agent_runtime.py:2180-2212`, `src/agent_runtime.py:2387-2403`).

For claude-anyteam this is primarily an anti-model for §1: do not replace Codex/Gemini/Kimi/Claude native processes with a lowest-common-denominator Python loop. ClawCode’s plugin and virtual-tool surface is extensible, but it preserves neither harness-native approval flows nor native tool/event semantics; it rebuilds them.

## §2 (visibility parity) comparison
ClawCode does have a good local visibility spine. The runtime records `tool_start`, `tool_delta`, permission denial, and `tool_result` events around each tool call (`src/agent_runtime.py:852-866`, `src/agent_runtime.py:950-973`, `src/agent_runtime.py:1022-1057`), and the bash tool streams stdout/stderr deltas with final exit-code metadata (`src/agent_tools.py:2896-3018`). The GUI returns transcript, usage, cost, stop reason, and errors from `/api/chat` (`src/gui/server.py:650-700`, `src/gui/server.py:715-725`), and its frontend renders collapsible tool cards from assistant/tool transcript entries so users see what the agent did, not just the final reply (`src/gui/static/app.js:307-333`, `src/gui/static/app.js:543-575`, `src/gui/static/app.js:751-754`).

It falls short of claude-anyteam’s visibility parity because the “lead” view is a local GUI/CLI result, not a live supervisor seeing peer subprocess events/prose/errors as native Claude would. `/api/chat` serializes `transcript`, not the full `events` stream (`src/gui/server.py:715-725`), and it returns after `agent.run()` completes in a background thread (`src/gui/server.py:669-689`), so tool deltas are captured in state but not delivered as a real-time peer visibility stream. The parity checklist also acknowledges missing rich TUI/interactive rendering and remote/team parity (`PARITY_CHECKLIST.md:568-593`, `PARITY_CHECKLIST.md:595-613`).

## §3 (peer efficiency) comparison
The peer-DM mechanism is only local message recording. `team_list`/`team_get`/`team_create`/`send_message` are registry tools (`src/agent_tools.py:809-887`), but `_send_message` just checks write permission, calls `TeamRuntime.send_message`, and returns the JSON state path (`src/agent_tools.py:2416-2451`). `TeamRuntime.send_message` appends a `TeamMessage` to an in-memory tuple and persists `.port_sessions/team_runtime.json` with `write_text` (`src/team_runtime.py:11`, `src/team_runtime.py:64-72`, `src/team_runtime.py:231-253`, `src/team_runtime.py:321-330`). The prompt guidance explicitly frames this as recording a handoff/note in local persisted history (`src/agent_prompting.py:355-364`), and the parity checklist says real team collaboration is missing (`PARITY_CHECKLIST.md:606-613`). RTT is therefore “whenever another local actor reads the file/GUI tab,” not a live mailbox with recipient wakeup, typed message kinds, or steer.

Capability-discovery latency has a notable trap: every stdio MCP resource/tool request opens a fresh `_StdioMCPConnection`, `Popen`s the server, initializes it, and waits up to 10 seconds for responses (`src/mcp_runtime.py:655-698`, `src/mcp_runtime.py:798-807`). Nested delegation has dependency metadata and batch summaries, but “parallel” is normalized to topological batching and batches are still executed serially through `for subtask in batch:` (`src/agent_runtime.py:2303-2311`, `src/agent_runtime.py:2469-2472`, `src/agent_runtime.py:2698-2752`). These are useful cautionary examples for §3: do not let local state recording, serial subtask loops, or per-call discovery masquerade as efficient peer coordination.

## Lessons we should adopt
1. **Render tool visibility as first-class cards, but from our native event envelope.** ClawCode’s GUI card pattern (`src/gui/static/app.js:307-333`, `src/gui/static/app.js:543-575`) is a compact way to show tool name, args, result, and error state. Target: use `src/claude_anyteam/messages.py:227-283` (`VisibilityEvent`) plus `src/claude_teams/server.py:710-797` (`check_teammate`) to return/render recent `tool_event` cards alongside prose/status, instead of only tmux output or inbox messages.
2. **Keep raw backend streams while deriving stable tool-event summaries.** ClawCode records both transcript tool messages and explicit event metadata (`src/agent_runtime.py:852-866`, `src/agent_runtime.py:1037-1057`); our current design is stronger because `src/claude_anyteam/headless_visibility.py:104-191` derives per-tool envelopes while preserving raw backend events, and emits them before terminal completion at `src/claude_anyteam/headless_visibility.py:372-392`. Target: continue applying this to Codex/Gemini/Kimi; avoid summarizing away backend-native fields.
3. **Adopt delegation/batch summary shape, not its serial executor.** ClawCode’s delegated-agent result includes child session ids, per-child stop reasons, batch status, dependency skips, and final output (`src/agent_runtime.py:2523-2633`). Target: when `src/claude_teams/server.py:300-339` spawns real teammates or task groups, mirror this summary in task metadata / visibility events so the lead can audit multi-peer batches without reading every pane.
4. **Cache capability discovery; do not probe on every call.** ClawCode’s per-request MCP stdio initialization is exactly the latency we want to avoid (`src/mcp_runtime.py:655-698`, `src/mcp_runtime.py:798-807`). Target: keep using eager Agent Card cache loading/invalidation in `src/claude_anyteam/capability_manifest.py:111-177`, and make any wrapper tool discovery read that cache instead of starting peer/harness probes during the task turn.
5. **Expose permission/tool errors as structured visibility, not prose-only.** ClawCode tags permission denials with event metadata (`src/agent_runtime.py:1022-1036`) and tool execution errors with `error_kind` (`src/agent_tools.py:90-111`). Target: ensure permission/request failures in `src/claude_teams/server.py:350-427` and routed harness wrappers become `turn_warning`/`tool_event` payloads in `src/claude_anyteam/messages.py:227-283`, not just free-form final text.

## Anti-patterns to avoid
1. **Do not build a universal agent loop that flattens harnesses.** ClawCode’s core design is a Python Claude Code clone over OpenAI-compatible tool calls (`README.md:91-97`, `src/openai_compat.py:250-271`); claude-anyteam’s §1 requires native harness preservation instead.
2. **Do not label local append-only notes as peer-DM.** ClawCode’s `send_message` records `.port_sessions/team_runtime.json` without live recipient delivery, locks, inbox semantics, or typed protocol handling (`src/team_runtime.py:231-253`, `src/team_runtime.py:321-330`); claude-anyteam should preserve mailbox semantics and low-RTT peer steering.
3. **Do not advertise capabilities before they are real.** The `Agent` schema advertises `run_in_background` and `isolation="worktree"` (`src/agent_tools.py:1107-1176`), while execution ignores those fields and only creates in-process children; the README also advertises “parallel” delegation (`README.md:724-739`) even though execution is serial inside each batch (`src/agent_runtime.py:2303-2311`, `src/agent_runtime.py:2469-2472`). Capability manifests in claude-anyteam must be conservative and test-backed.
4. **Do not pay per-turn/per-tool capability startup costs.** Fresh MCP `Popen` + initialize per resource/tool request (`src/mcp_runtime.py:655-698`) is a direct §3 latency anti-pattern.
