# Unified architectural comparison matrix — six research digests

Source digests: [clawcode](clawcode.md), [maorinka-claude-rs](maorinka-claude-rs.md), [piebald-system-prompts](piebald-system-prompts.md), [nwyin-cleanroom](nwyin-cleanroom.md), [pickle-pixel-hydrateams](pickle-pixel-hydrateams.md), [aproto-codex-bridge](aproto-codex-bridge.md).

North-star shorthand:
- **§1** = preserve each teammate harness' native capabilities instead of flattening to one generic agent loop.
- **§2** = lead/peer visibility parity: model/tool/prose/errors visible as typed substrate events, not just pane text.
- **§3** = peer efficiency: low-latency DM/wakeup, cheap capability discovery, low-noise idle/heartbeat.

## Convergent findings (cross-source corroborated)

- **SendMessage visibility invariant — confirmed by piebald + nwyin.** Piebald's extracted tool description says plain text is not visible to other agents; nwyin's cleanroom repeats the same team prompt invariant and models `SendMessage` as the peer-communication tool. Our position: this belongs in every routed backend prompt and in the `send_message` tool description; final/prose output is never a DM.

- **Event-driven inbox wakeup — confirmed by aproto-codex-bridge + maorinka-claude-rs.** aproto has the concrete file-system implementation (`fs.watch`, 100ms active wake, 5s fallback). Maorinka supplies the durable supervisor shape (`tokio::mpsc`, `oneshot`, cancellation tokens, pending request maps). Our position: implement a Python watcher first, keep poll as fallback, and keep the API narrow enough that a Rust sidecar remains possible.

- **Pre-cache / pre-hydrate capability manifests at startup — confirmed by HydraTeams, maorinka, nwyin, and clawcode by contrast.** HydraTeams wins by ensuring Claude's native tools are in the teammate body from the first LLM round; maorinka's MCP manager shows bounded concurrent prewarm; nwyin uses prompt attachments/deferred-tool deltas; clawcode shows the latency trap of per-call MCP stdio startup. Our position: wrapper MCP tools and rich Agent Card manifests must be mounted, loaded, and sanity-checked before declaring a teammate available.

- **Typed protocol envelopes beat prose lifecycle — confirmed by nwyin + maorinka + aproto.** Nwyin models shutdown/plan replies as `SendMessage` payload variants and has named `TeammateIdle`/`TaskCompleted` hooks. Maorinka's prose-ish mailbox and split substrates are a cautionary failure. aproto's `idle_notification` JSON is directionally right but not a versioned message kind everywhere. Our position: keep expanding `messageKind`/typed lifecycle payloads; do not ask peers to parse prose for lifecycle.

- **Dedicated tool surface beats shell/prose fallback — confirmed by piebald + aproto + HydraTeams.** Piebald says use a dedicated SendMessage tool; HydraTeams keeps native Claude tools in the body; aproto's `codex-bridge send` works but lacks typed schemas, errors, and tool-call visibility. Our position: wrapper MCP is the right primitive for routed backends; shell senders are emergency fallback only.

- **Structured event data and UI projection are complementary — confirmed by clawcode + aproto + maorinka + nwyin.** Clawcode/aproto have better visible cards/pane UX than us; maorinka/nwyin define useful event taxonomies. Our position: do not replace our `VisibilityEvent` envelopes; build card/TUI projections on top of them.

- **Capability declarations must be conservative and test-backed — confirmed by clawcode + maorinka + aproto + piebald.** Clawcode advertises fields whose executor ignores them; maorinka has unregistered SendMessage/schema drift; aproto infers capability from name prefixes; Piebald emphasizes exact schemas and contexts. Our position: a manifest flag is a promise; no flag without a runtime path and test.

- **Bound long payloads before they enter inbox/model context — confirmed by aproto + nwyin.** aproto stores full results in `results/` and sends only previews; nwyin treats attachments as structured prompt deltas. Our position: add a first-class attachment/preview protocol for long peer DMs, errors, and task outputs.

## Divergent findings

- **Hydrate native Claude body vs preserve native external harnesses.** HydraTeams gives native Agent Teams speed/visibility by using Claude Code as the body for every teammate. That is compelling for §3 but violates our §1 moat: Codex is no longer Codex App Server, Gemini is no longer ACP, Kimi is no longer its own CLI. **Our position:** adopt pre-hydration/startup preflight, not the full brain-swap architecture.

- **One reimplemented universal loop vs adapter-per-harness.** Clawcode, maorinka, and nwyin all centralize around one rebuilt Claude-like loop. That gives control over schemas/events, but loses external harness-specific affordances. **Our position:** use their schema/event discipline as library patterns; do not migrate coordination into one generic Python/Rust model loop.

- **CLI protocol vs model-visible MCP vs native SendMessage.** aproto exposes `codex-bridge send` through shell; native Claude/Piebald/HydraTeams rely on tool schemas; claude-anyteam uses wrapper MCP. **Our position:** keep model-visible wrapper MCP as the standard because it supports exact schema, typed errors, visibility events, and membership validation. CLI is acceptable only as a last-ditch break-glass path.

- **Busy-turn steer everything vs `messageKind` gates.** aproto unconditionally injects active-turn inbound messages as `turn/steer`, optimizing reactivity but perturbing active work and mislabeling peer DMs as leader messages. Nwyin's typed lifecycle union and our recent `messageKind` work favor semantic routing. **Our position:** default peer messages to informational; only manifest-authorized `kind=steer` may interrupt an active turn.

- **Continuous worker thread vs task-scoped/forked lineage.** aproto's one App Server session preserves continuity naturally and supports native TUI attachment; claude-anyteam isolates task lineage with explicit `thread/fork` and materialization fallback. **Our position:** keep task isolation as default; optionally add a continuous-session capability only if it is explicitly declared and bounded by context controls.

- **Deferred tool loading vs eager critical tools.** Piebald/nwyin show ToolSearch can work for a large native corpus. Our routed backends have already hallucinated missing `send_message`. **Our position:** noncritical tools may be deferred/discovered; `send_message`, `task_update`, `read_config`, `read_inbox`, `task_list`, and capability lookup must be eagerly exposed and repeatedly named.

- **Pane/TUI visibility vs structured event substrate.** aproto/clawcode are more pleasant live; our event log is more queryable and lead-safe. **Our position:** build their UX as projection, not replacement. The canonical record remains typed visibility events.

## Honest comparison: where our architecture is better, where it isn't

Where claude-anyteam is better:
- **Heterogeneous harness preservation.** Clawcode/maorinka/nwyin rebuild one Claude-ish loop; HydraTeams uses one Claude body. Our adapter-per-harness design is still the only architecture here that keeps Codex App Server, Gemini ACP, Kimi, and Claude-style teammates as distinct capability surfaces.
- **Typed wrapper MCP protocol surface.** Compared with aproto's shell CLI and clawcode's local note tools, our `send_message`/`task_update`/`read_config`/manifest tools have schema, membership validation, tool-call visibility, and direct errors.
- **Capability manifests.** aproto has prefixes, HydraTeams has fixed Claude tools, Piebald/nwyin describe tool capabilities but not per-peer harness primitives. Our rich Agent Card path is the right abstraction when it is eagerly cached and test-backed.
- **Peer-DM vs steer semantics.** Our `messageKind`/manifest-gated steer direction is safer than aproto's “steer all active-turn messages.”
- **Structured lead visibility substrate.** Our `VisibilityEvent` envelopes are more queryable than aproto/claw pane-only UX and more applicable to foreign harnesses than native Claude-only streams.
- **Explicit task lifecycle.** We should keep explicit task IDs/status updates; aproto's all-owned auto-complete is unsafe.

Where claude-anyteam is not better:
- **Inbox wake latency.** aproto's `fs.watch` loop is plainly better than our fixed `poll_interval_s` sleeps. Adopt it wholesale with fallback.
- **Live UI polish.** Clawcode's tool cards and aproto's ANSI/Ink/native TUI make agent activity legible. We have better data, but the operator experience is worse.
- **Native hot-path speed.** HydraTeams gets SendMessage/TaskUpdate inside the Claude process from round one. We can match only by making wrapper MCP and manifests pre-mounted/preflighted; lazy discovery will keep losing.
- **Transport recovery.** aproto's WebSocket/TUI restart logic is more mature than our current classify-and-report posture.
- **Lifecycle schema completeness.** Nwyin's plan/shutdown/idle/task-complete typed hooks are more complete as a conceptual model than our still-mixed prose/JSON traffic.
- **Async supervision.** Maorinka's Rust architecture shows a cleaner resident supervisor/channel/cancellation model than our sleep-loop adapters.
- **Long-output context hygiene.** aproto's result-file + preview invariant is simple and effective; we need a general attachment/preview contract.

Adopt wholesale, not just “inspired by”:
- aproto-style **watcher with active/fallback polling**.
- Piebald/nwyin **plain text is not delivered; call `send_message`** wording.
- aproto-style **bounded preview + stored artifact** for long results.
- Clawcode/aproto **tool-event cards** as a UI projection, with our event log as source.
- HydraTeams **startup compatibility/preflight checklist** mindset, without the model spoofing.

## Post-session deltas

Patterns identified during the 2026-04-28 implementation swarm that were not fully represented in the original digest comparison:

- **Recurring 900s App Server turn-timeout reap.** Multiple Codex teammates doing multi-file work were killed at the App Server turn boundary before all intended commits landed. #45 closed the most important operational gap by adding `checkpoint_commit`, prompt guidance to use it liberally, configurable timeout plumbing tests, and an end-to-end checkpoint-before-timeout regression (`87c32a0`/`1fd1fe2`/`436854d`/`accc001`).
- **Duplicate-schema-commit incident.** The apparent duplicate `schema: add inbox attachment metadata` commit was traced to a cherry-pick from the `long-output` branch, not direct substrate/index bleed. #48 still turned the incident into defensive isolation: explicit-cwd teammate spawns that collide with an existing Git worktree now get a real per-teammate worktree, with audit/report/fix/test commits (`9f66d8d`/`8083fe5`/`e5a4469`).
- **Persistent SendMessage tool-discovery flap.** #27 and #32 made the invariant and exact tool names visible, but several Codex sessions still emitted “I don't have `send_message`” prose. #44 shipped wrapper-MCP diagnostics at `2900940`; #51 is the in-flight root-cause-and-fix task using those traces.
- **Codex CLI empty `task_complete` auto-emits.** Several parked/idle Codex turns produced `task_complete` payloads with empty `files_changed` at turn end. Treat these as operational artifacts unless git status and integration HEAD prove substantive work landed.

## What we learned about the swarm

- **Incremental-commit discipline is mandatory for Codex teammates.** The session repeatedly showed that single large Codex turns are fragile under the 900s reap pattern; small commits plus `checkpoint_commit` preserve progress and make recovery cheap.
- **Salvage-from-worktree is a first-class recovery move.** #28 and #41 both benefited from recovering uncommitted or partially committed work after a teammate was reaped, then finishing the tests/merge from a clean lane.
- **FF-merge verification is part of task completion, not postscript.** Codex `task_complete` messages can be premature or empty; completion requires verifying that integration HEAD contains the expected commit(s) and that the worktree status is understood.
- **Idle noise is not a substantive checkpoint.** Repeated `idle_notification`, shutdown responses, and empty turn-end payloads should be separated from real checkpoint commits, validation outputs, and DMs that report landed SHAs.

## Open questions / future research lanes

- What are actual end-to-end peer-DM RTT distributions after watcher adoption: idle recipient, busy recipient, same backend, cross-backend, NFS/macOS/Linux?
- Should evented inbox be pure Python (`watchfiles`/`watchdog`), direct stdlib polling with `mtime`, or a minimal Rust `notify` sidecar?
- What is the exact typed schema for lifecycle payloads (`shutdown_request`, `shutdown_response`, `plan_approval`, `idle_notification`, `task_complete`, `handoff`) and how does it coexist with plain peer prose?
- Should `send_message` support attachments/previews directly, or should `protocol_io` auto-spill long bodies to artifacts before calling it?
- Can we expose a native Codex TUI/WebSocket view while preserving wrapper MCP tool visibility and structured event logging?
- How do we prevent capability cache staleness after teammate restart, manifest delete, or backend degradation without adding per-message discovery RTT?
- Should critical protocol tools be non-deferred for every backend even if future ToolSearch/list-protocol-tools exists?
- Can a “transport avatar” preserve HydraTeams-like native Agent Teams hot-path speed without replacing Codex/Gemini/Kimi harness bodies?
- What policy should govern continuous long-lived thread mode vs task-scoped `thread/fork` isolation?
